# 삭제 없이 불편 해소 — Web2024 비NIA 코드 대안

> 작성일: 2026-04-07
> 목적: 비NIA 코드(IPMS, DataHub, Demo)를 삭제하지 않고 검색 오염과 소스 로딩 느림 문제를 해결하는 방안

---

## 문제 정리

| # | 불편 사항 | 원인 |
|---|----------|------|
| 1 | **검색 시 비NIA 코드가 함께 검색됨** | IDE/에디터가 전체 프로젝트를 검색 범위로 잡음 |
| 2 | **파일이 무거워 소스 로딩이 느림** | IPMS 640개 Java 파일 등 불필요한 파일을 인덱싱함 |

두 문제의 공통 근본 원인: **IDE가 비NIA 디렉토리를 인덱싱하고 있음**

---

## 대안 1: IDE에서 디렉토리 제외 (가장 쉬움)

### 난이도: ★☆☆ (5분)

### VS Code

`.vscode/settings.json`에 아래 설정을 추가하면 **검색, 파일 탐색, 인덱싱** 모두에서 제외된다.

```json
{
  "files.exclude": {
    "**/BE/app-ipms": true,
    "**/BE/app-dataHub": true,
    "**/BE/app-demo": true,
    "**/FE/src/views-ipms": true,
    "**/FE/src/views-dataHub": true,
    "**/FE/src/views-aiTemplate": true,
    "**/FE/src/router/ipms": true,
    "**/FE/src/router/dataHub": true,
    "**/FE/src/router/aiTemplate": true
  },
  "search.exclude": {
    "**/BE/app-ipms": true,
    "**/BE/app-dataHub": true,
    "**/BE/app-demo": true,
    "**/FE/src/views-ipms": true,
    "**/FE/src/views-dataHub": true,
    "**/FE/src/views-aiTemplate": true
  }
}
```

**효과**:
- 검색(Ctrl+Shift+F)에서 비NIA 코드 완전 제외
- 파일 탐색기에서 비NIA 폴더 숨김
- **파일 워처(file watcher) 부하 감소** → 로딩 속도 개선

**장점**: 코드 변경 없음, git에 영향 없음, 언제든 해제 가능
**단점**: 팀원마다 설정 필요 (`.vscode/`를 커밋하면 공유 가능)

### IntelliJ IDEA

1. 프로젝트 뷰에서 `BE/app-ipms` 우클릭 → **Mark Directory as → Excluded**
2. 같은 방식으로 `app-dataHub`, `app-demo`, `views-ipms`, `views-dataHub`, `views-aiTemplate` 제외

또는 `.idea/` 설정 파일에서 직접 수정 가능.

**효과**: 인덱싱 제외 → 검색 제외 + 로딩 속도 개선

## 대안 2: .gitignore 활용 — 로컬 전용 제외

### 난이도: ★☆☆ (2분)

`.git/info/exclude`를 수정하면 **로컬에서만** 특정 파일을 무시할 수 있다 (공유 `.gitignore`에 영향 없음).

```bash
# .git/info/exclude (로컬 전용, 커밋되지 않음)
BE/app-ipms/
BE/app-dataHub/
BE/app-demo/
FE/src/views-ipms/
FE/src/views-dataHub/
FE/src/views-aiTemplate/
```

**주의**: 이 방법은 **이미 추적 중인 파일에는 적용되지 않는다**. git이 이미 추적하고 있으므로 ignore가 동작하지 않는다. 단, IDE의 git 통합에서 해당 파일의 변경을 표시하지 않는 효과는 있을 수 있다.

→ 따라서 **이 방법은 권장하지 않음**. 대안 1(IDE 제외)이 더 확실하다.

## 대안 3: VS Code 멀티루트 워크스페이스

### 난이도: ★★☆ (10분)

Web2024가 이미 `.code-workspace` 파일을 가지고 있으므로, NIA 관련 폴더만 워크스페이스에 등록하는 방법.

`Web2024_NIA.code-workspace` (새로 생성):

```json
{
  "folders": [
    { "path": "BE/app-nia", "name": "BE - NIA App" },
    { "path": "BE/common-base", "name": "BE - Common Base" },
    { "path": "BE/common-web", "name": "BE - Common Web" },
    { "path": "BE/common-mq", "name": "BE - Common MQ" },
    { "path": "BE/common-ws", "name": "BE - Common WS" },
    { "path": "FE", "name": "Frontend" }
  ],
  "settings": {
    "files.exclude": {
      "src/views-ipms": true,
      "src/views-dataHub": true,
      "src/views-aiTemplate": true,
      "src/router/ipms": true,
      "src/router/dataHub": true,
      "src/router/aiTemplate": true
    }
  }
}
```

**효과**:
- BE에서 app-ipms, app-dataHub, app-demo가 완전히 보이지 않음
- FE는 하나로 열되 내부 제외 설정 적용
- 검색 범위가 워크스페이스 폴더로 자동 한정

**장점**: 가장 확실한 분리, 파일 탐색기가 깔끔
**단점**: 프로젝트 루트 기준 경로가 바뀌어 터미널 작업 시 혼란 가능

## 대안 4: Gradle 빌드에서 비NIA 모듈 비활성화

### 난이도: ★☆☆ (1분)

`BE/settings.gradle`에서 비NIA 모듈을 주석 처리하면 **빌드 시 해당 모듈을 아예 스킵**한다.

```gradle
rootProject.name = 'web'
include 'common-base'
include 'common-web'
include 'common-mq'
include 'common-ws'
// include 'app-datahub'
include 'app-nia'
// include 'app-demo'
// include 'app-ipms'
```

**효과**:
- `./gradlew clean build`가 NIA만 빌드 → **빌드 시간 대폭 단축**
- IntelliJ에서 Gradle sync 시 비NIA 모듈이 프로젝트에서 빠짐 → **인덱싱 감소**

**장점**: 빌드 속도 + IDE 인덱싱 동시 해결
**단점**: 공유 파일 1개(settings.gradle) 수정 필요 → merge 시 3줄 충돌 가능 (난이도 매우 낮음)

## 대안 5: FE 개발서버를 NIA 모드로만 실행

### 난이도: ★☆☆ (이미 되어 있을 수 있음)

FE 개발 시 `.env.nia` 환경으로만 실행하면, webpack이 NIA 외 코드를 번들에 포함하지 않는다.

```bash
npm run build:nia    # 빌드 시 NIA 모드
```

개발서버(dev)도 NIA 모드로 실행:

```bash
# .env 파일 수정 또는 환경변수 지정
VUE_APP_PROJECT=nia npm run dev
```

**효과**: 런타임에서 비NIA 코드가 로드되지 않음. 단, IDE 인덱싱에는 영향 없음.

## 추천 조합

**즉시 적용 (코드 변경 없음, 5분)**:
1. **대안 1** (VS Code `files.exclude` + `search.exclude`) → 검색 오염 해결 + 로딩 개선
2. **대안 5** (NIA 모드 개발서버) → FE 빌드 속도 개선

**추가로 하면 좋은 것 (코드 3줄 변경, 1분)**:
3. **대안 4** (Gradle 모듈 비활성화) → BE 빌드 속도 + IntelliJ 인덱싱 개선

이 조합이면 **소스 삭제 없이** 검색 오염과 로딩 느림 문제가 모두 해결된다.
재결합 시에도 settings.gradle 3줄 주석 해제만 하면 되므로 merge 리스크가 거의 없다.
