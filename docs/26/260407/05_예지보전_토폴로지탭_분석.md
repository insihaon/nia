> 대상 화면: PredictiveMaintenanceTicket.vue → Map2D.vue (토폴로지 우측 상단 탭)

## 1. 각 탭이 보여주는 것

토폴로지 상단의 3개 탭은 **광 캐리어(trunk)에 포함된 각 노드(ROADM 장비)의 광 성능 지표**를 전환 표시하는 기능이다.

| 탭 | 표시 값 | 의미 | 예시 |
|----|---------|------|------|
| **Rx / Tx** | 수신/송신 광 레벨 | 각 노드에서 측정한 광 입력(Rx)/출력(Tx) 파워 (dBm) | `-20.5 / -18.3` |
| **Span Loss / Node Gain** | 구간 손실 / 노드 이득 | 노드 간 광섬유 구간의 손실량과 노드 내 증폭 이득량 (dB) | `3.2 / 2.1` |
| **Node Total Deviation** | 노드 총 편차 | 해당 노드의 광 레벨이 기준값에서 얼마나 벗어났는지 (dB) | `1.5` |

### 광 전송망에서의 역할

```
[노드A] ──── 광섬유 구간 ──── [노드B] ──── 광섬유 구간 ──── [노드C]
  Tx=-18.3                       Rx=-20.5                      Rx=-21.2
                Span Loss=2.2              Span Loss=0.7
         Node Gain=2.0             Node Gain=1.8
         NTD=0.5                   NTD=1.5  ← 편차가 크면 장애 예측
```

- **Rx/Tx**: 각 노드의 "현재 광 파워" — 기본 모니터링 지표
- **Span Loss/Node Gain**: 구간별 "손실과 보상" — 광섬유 열화나 장비 이상 판단
- **NTD**: 누적 편차 — 값이 ±2 이상이면 **조치 필요 구간**으로 판정 (전표 발행 기준)

## 2. 동작 원리 — 데이터 흐름 (수정)

### 전체 흐름

```
[PostgreSQL mba 스키마]
  tb_roadm_optical_performance_hist (실시간 수집 데이터)
  tb_roadm_optical_performance_daily (일별 집계 - span_loss, node_gain, ntd)
      ↓
[SQL: SELECT_PMM_TOPOLOGY_LIST]
  RX_TX = "수신값/송신값"
  SPAN_GAIN = "손실값/이득값"
  NODE_TOTAL_DEVIATION = 편차값
      ↓
[PostgreSQL → 소문자 변환]
  따옴표 없는 alias는 PostgreSQL이 자동 소문자 처리
  → rx_tx, span_gain, node_total_deviation
      ↓
[Map2D 라이브러리 — 'mba' converter]
  ★ 핵심: converter가 키를 재매핑한다
  rx_tx        → rx_tx       (유지)
  span_gain    → span_gain   (유지)
  node_total_deviation → ntd (축약 매핑!)
      ↓
[탭 전환]
  node[선택된_탭_label]로 접근
  'rx_tx' → node.rx_tx
  'span_gain' → node.span_gain
  'ntd' → node.ntd  ← converter가 매핑한 키
```

### mba converter 내부 (map2d_untact.min.js)

```javascript
// 라이브러리가 API 데이터를 내부 노드 객체로 변환할 때:
{
  rx_tx: t.rx_tx,                           // 그대로 유지
  span_gain: t.span_gain,                   // 그대로 유지
  ntd: t.node_total_deviation.toFixed(1),   // ★ node_total_deviation → ntd로 매핑
  ntdDiff: r,                               // NTD 차이값도 계산
  // ...
}
```

### 탭 전환 시 핵심 코드

```javascript
// Map2D.vue — watch (수정 후)
nodeInfoOption(newValue) {        // newValue = 'rx_tx' | 'span_gain' | 'ntd'
  this.map.options.node_info_field = newValue
  this.map.options.updateNodeInfoDisplayField()  // ★ map이 아닌 map.options의 메서드
}

// map2d_untact.min.js — 내부 동작
updateNodeInfoDisplayField: function() {
  let field = options.node_info_field
  nodes.select("text").text(function(node) {
    return node[field]    // node['ntd'] → "1.5"
  })
}
```

## 3. 발견된 버그와 수정 내역

### 버그 1: updateNodeInfoDisplayField 호출 대상 오류 (기존 버그)

```javascript
// 수정 전 (Map2D.vue) — 에러 발생
this.map.updateNodeInfoDisplayField()  // map 인스턴스에는 이 메서드가 없음

// 수정 후
this.map.options.updateNodeInfoDisplayField()  // options 객체에 바인딩된 메서드
```

`updateNodeInfoDisplayField`는 Map2D 라이브러리에서 `map` 인스턴스가 아닌 `options` 객체에 정의되어 있다. 기존 코드에서 `map`에서 호출했기 때문에 탭 전환 시 항상 에러가 발생했고, 결과적으로 **모든 탭에서 Rx/Tx 값이 그대로 유지**되는 현상이 발생했다.

### 버그 2: d3.js의 undefined 처리 특성

d3의 `.text(undefined)`는 **기존 텍스트를 변경하지 않는다**. 따라서 메서드 호출 에러 → catch 없이 실패 → 텍스트 갱신 안 됨 → Rx/Tx 값 유지.

### NTD label은 `ntd`가 맞음 (초기 진단 수정)

초기 분석에서 "DB 키 `node_total_deviation`과 탭 label `ntd`가 불일치"라고 판단했으나, 이는 **오진**이었다.

mba converter가 `node_total_deviation`을 `ntd`로 매핑하고 있어 label `ntd`가 정확하다. 라이브러리의 dat.gui 설정에서도 `["rx_tx","span_gain","ntd"]`로 등록되어 있다.

## 4. Span Loss / Node Gain 탭 데이터 부재

DB 직접 조회 결과:

```json
{
  "rx_tx": "-2.9/",
  "span_gain": "/",
  "node_total_deviation": "0"
}
```

`span_gain` 값이 `"/"`인 이유:
- SQL에서 `LEFT JOIN`으로 daily 테이블을 조인하는데, **daily 테이블에 데이터가 없으면** `span_loss`와 `node_gain`이 NULL
- `CONCAT(NULL, '/', NULL)` → `"/"`

**이 탭이 정상 동작하려면 `tb_roadm_optical_performance_daily` 테이블에 데이터가 적재되어야 한다.** 광 PM 엔진(`Engine_nia_optical_pm`)이 daily 집계를 수행하는지 확인 필요.

## 5. 이 시스템의 출처와 용어 호환성

### MBA/ROADM 시스템 출처

이 예지보전 기능은 **MBA(Multiservice Broadband Architecture)** 시스템에서 가져온 것으로, 원래 ROADM 기반 광 전송망 장비를 관리하던 시스템이다.

| 항목 | 내용 |
|------|------|
| DB 스키마 | `mba` |
| 테이블 접두어 | `tb_roadm_` |
| 엔진 프로젝트 | `Engine_nia_optical_pm` (광 PM 수집 엔진) |
| Java 패키지 | `kr.go.ap.core.primary.nia.entity.mba.performance` |

### 광 전송망 용어 호환성

Rx/Tx, Span Loss, Node Gain, NTD는 **광 전송망 표준 용어**이다. ROADM 장비라면 제조사(삼성, 화웨이, 노키아 등)와 무관하게 동일한 의미를 가진다.

| 용어 | 표준 여부 | 설명 |
|------|----------|------|
| Rx (Receive Power) | ITU-T G.697 | 광 수신 파워 — 모든 광 장비 공통 |
| Tx (Transmit Power) | ITU-T G.697 | 광 송신 파워 — 모든 광 장비 공통 |
| Span Loss | ITU-T G.680 | 노드 간 광섬유 구간 손실 — DWDM/ROADM 표준 |
| Node Gain | ITU-T G.680 | 노드 내 광 증폭 이득 — EDFA/ROADM 표준 |
| Node Total Deviation (NTD) | 벤더 확장 | 기준값 대비 누적 편차 — MBA 시스템 자체 정의일 가능성 |

**결론**: Rx/Tx, Span Loss, Node Gain은 **광 전송망 표준 지표**이므로 NIA 시스템에서도 동일한 의미로 사용 가능하다. NTD는 MBA 시스템의 자체 정의 지표일 수 있으나, 광 PM 엔진(`Engine_nia_optical_pm`)에서 이미 `node_total_deviation` 컬럼을 계산하고 있으므로 NIA에서도 유효하다.
