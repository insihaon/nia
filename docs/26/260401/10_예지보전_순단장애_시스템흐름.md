## 2. 예지보전(PM) 상세 흐름

### 2.1 엔진 처리 흐름

#### Phase 1: 데이터 수집 (Linkage)

| 단계 | 서비스 | 동작 |
|------|--------|------|
| 1 | `EmsMmcHdlService` | EMS에 TL1 프로토콜로 접속 (61.252.53.34:42001) |
| 2 | | `RTRV-SIPC` 명령 → 장비 쉘프 구성 수집 |
| 3 | | `RTRV-PM` 명령 → 광성능 데이터 수집 (RX/TX) |
| 4 | `ParsingHdlService` | TL1 응답 파싱, DB 저장 |

**저장 테이블**: `NIA.TB_PERFORMACE_MST` (SYSNAME, PORT, RXCUR, TXCUR, OCRTIME)

#### Phase 2: 일일 집계 (스케줄러, 매일 01:05)

```
PmDailyService.performanceDailyHdl()
  ├─ FC_SET_PERFORMANCE_DAILY()         → 일별 광성능 집계
  ├─ FC_SET_PERFORMANCE_DAILY_NTD()     → NTD/Span Loss/Node Gain 계산
  ├─ FC_SET_PERFORMANCE_DAILY_REFERENCE() → 기준값 산출
  └─ AiPmDataSendService.sendPmData()  → AI 모델 서버에 데이터 전송
```

**저장 테이블**: `MBA.TB_ROADM_OPTICAL_PERFORMANCE_DAILY`

| 필드 | 설명 |
|------|------|
| TRUNK_NAME | 캐리어명 (예: SEOUL--BUSAN) |
| DIRECTION | 방향 (UP/DOWN) |
| SYSNAME | 장비명 |
| VALUE_MAX | 최대 광레벨 |
| RX_CUR / TX_CUR | 수신/송신 현재값 |
| NODE_TOTAL_DEVIATION | 노드 전체 편차 (핵심 지표) |
| SPAN_LOSS | 구간 손실 |
| NODE_GAIN | 노드 이득 |

#### Phase 3: 전표 생성 (스케줄러, 매일 02:05)

```
PmTicketService.ticketHdlProcessor()
  ├─ TicketCaseOneService: NTD 변화량 기반 전표
  │   ├─ 14일간 NTD 이력 조회
  │   ├─ 판단: |NTD_VARIANCE| > MAX(2.0dB, 3σ)
  │   └─ 전표 생성 or 기존 전표 갱신
  │
  └─ PmWarningDateService: AI 예측 경보 기반 전표
      └─ AI 모델 warning_date 임계값 확인
```

**전표 생성 판단 기준:**
- **Case 1**: NTD 변화량이 2.0dB 이상 또는 표준편차의 3배 이상
- **Case 2**: AI 모델이 경고 날짜를 예측한 경우

**저장 테이블**: `MBA.TB_ROADM_PREDICTIVE_MAINTENANCE_TICKET`

| 필드 | 설명 |
|------|------|
| TICKETNO | 전표 번호 |
| ISSUE_DATE | 발행일 |
| CLEAR_DATE | 마감일 (null이면 미해결) |
| TRUNK_NAME | 캐리어명 |
| DIRECTION | 방향 (UP/DOWN) |
| REASON | 사유 ("Node Total Deviation 변화" / "광레벨 장기 변화 예측 경보") |
| TARGET_SYSNAME | 영향 노드 (콤마 구분) |
| NODE_TOTAL_DEVIATION | NTD 값 |

## 3. 순단장애(MBA) 상세 흐름

### 3.1 엔진 처리 흐름

#### Phase 1: 저광레벨 감지

```
EmsMmcHdlService → ParsingHdlService → LineMonitoringHdlService
  └─ 감지 기준: |VALUE_MAX - VALUE_MIN| ≥ 3dB → 저광레벨 판정
```

#### Phase 2: 전표 생성

```
LineMonitoringHdlService
  └─ TicketDataService.setTicketData()
     ├─ UP/DOWN 방향별 분류
     ├─ 트렁크별 그룹핑
     └─ RabbitMQ → MbaEnginePrdAmqp
        └─ Exchange: nia.exchangeeEgineMbaTicketDirectly

MbaTicketMsgListener.onMessage()
  └─ TicketCreateService.createMbaTicket()
     ├─ 조건: 관련 경보 2건 이상
     ├─ 고유 ticketId 생성
     ├─ TB_MBA_TICKET_CURRENT 저장
     ├─ TB_MBA_TICKET_CABLE 저장 (A-Z 구간)
     └─ EngineToUiTicketPrdAmqp → WebSocket 알림
```

**저장 테이블**: `MBA.TB_MBA_TICKET_CURRENT`

| 필드 | 설명 |
|------|------|
| TICKET_ID | 전표 ID |
| TICKET_TYPE | "PF" (Performance Fault) |
| STATUS | INIT → ACK → FIN |
| ROOT_CAUSE_TYPE | "MomentaryBreakoff" |
| TICKET_RCA_RESULT_CODE | "OPTICAL_CABLE_LOW_SIGNAL" |
| TRUNK_NAME | 캐리어명 |
| DIRECTION | 방향 |
| FAULT_TIME | 장애 발생 시각 |

### 3.2 UI 처리 흐름

```
사용자가 "순단장애 감시" 화면 진입
  │
  ▼
① 전표 목록 로드 (SELECT_TICKET_CUR_LIST)
  │  └─ 조건: IS_MBA=true, MAX_DAYS=14
  │
  ▼
② WebSocket 실시간 수신 (채널: RCA_TICKET)
  │  ├─ 이벤트: MBA_NEW → 순단장애.mp3 재생
  │  └─ Vuex 스토어 업데이트 → 화면 즉시 반영
  │
  ▼
③ 전표 클릭 → 상세 패널
  │  ├─ 토폴로지 맵 로드 (topology2d/index.vue)
  │  │   ├─ SELECT_TICKET_ROOT_ALARM_LIST
  │  │   └─ SELECT_MBA_TOPOLOGY_LIST
  │  ├─ 경보 리스트 (SELECT_MBA_LOW_ALARM_LIST)
  │  │   └─ 심각도: Critical(-60dB) / Major(≥10dB) / Minor(≥3dB)
  │  └─ 영향회선 리스트
  │
  ▼
④ 전표 처리 (ModalRcaProcessHandling)
   ├─ 접수 (INIT → ACK)
   ├─ 완료 (ACK → FIN)
   └─ 일괄처리 (ModalRcaBatchProcessing)
```

**전표 상태 색상:**

| 상태 | 색상 | 설명 |
|------|------|------|
| INIT | 빨강 (#b14948) | 신규 발생 |
| ACK | 주황 (#F7AA17) | 접수 완료 |
| FIN | 초록 (#52A43A) | 처리 완료 |
| AUTO_FIN | 올리브 (#adcc1e) | 자동 완료 |
| NATURE | 다크올리브 (#70802b) | 자연 복구 |

## 4. PM vs MBA 비교

| 항목 | 예지보전 (PM) | 순단장애 (MBA) |
|------|-------------|--------------|
| **목적** | 장애 사전 예측 | 실시간 순단 감지 |
| **감지 방식** | NTD 추이 분석 + AI 예측 | 광레벨 급변 감지 (≥3dB) |
| **전표 생성** | 스케줄러 (매일 02:05) | 실시간 (즉시) |
| **응답 시간** | 시간~일 단위 | 초~분 단위 |
| **Root Cause** | NTD 변화 / AI 경보 | MomentaryBreakoff |
| **전표 테이블** | TB_ROADM_PREDICTIVE_MAINTENANCE_TICKET | TB_MBA_TICKET_CURRENT |
| **UI 갱신** | API 폴링 | WebSocket 실시간 |
| **경보음** | 없음 | 순단장애.mp3 |
| **처리 워크플로** | 검토의견 등록 | 접수(ACK) → 완료(FIN) |

## 7. 정상 동작 시 기대 결과

### 7.1 예지보전(PM) 정상 동작 시

**엔진 측:**
1. 매일 01:05 — 광성능 일일 집계 데이터 생성 (TB_ROADM_OPTICAL_PERFORMANCE_DAILY)
2. NTD, Span Loss, Node Gain 값이 계산되어 저장
3. AI 모델 서버에 데이터 전송, 예측 결과 수신
4. 매일 02:05 — NTD 변화량 |2.0dB 이상| 또는 |3σ 이상| 인 캐리어에 대해 전표 자동 생성

**UI 측:**
1. "광모듈 장애 예측" 화면에 미해결 전표 카드 목록 표시
2. 각 카드에 전표번호, 사유, 캐리어명, NTD값, 검토의견 건수 표시
3. 전표 더블클릭 시 해당 캐리어의 토폴로지 맵 렌더링
4. 라디오 버튼 전환 시:
   - **Rx/Tx**: 각 노드의 수신/송신 광레벨 (예: "-12.3/-5.1")
   - **Span Loss / Node Gain**: 구간 손실 / 노드 이득 값
   - **NTD**: 각 노드의 Node Total Deviation 값
5. AZ 구간(조치 필요 구간)이 NTD 차이 ≥ 2dB 기준으로 자동 산출
6. 검토의견 등록/수정/삭제 가능

### 7.2 순단장애(MBA) 정상 동작 시

**엔진 측:**
1. EMS에서 15분 주기로 광성능 데이터 수집
2. |VALUE_MAX - VALUE_MIN| ≥ 3dB 감지 시 즉시 전표 생성
3. WebSocket으로 UI에 실시간 알림 전송

**UI 측:**
1. "순단장애 감시" 화면에 최근 14일 전표 목록 표시
2. 신규 전표 발생 시 경보음(순단장애.mp3) 재생 + 실시간 화면 갱신
3. 전표 클릭 시:
   - 토폴로지 맵에 장애 구간(A-Z) 하이라이트
   - 경보 리스트에 심각도별(Critical/Major/Minor) 색상 표시
   - 영향회선 리스트 표시
4. 전표 상태 워크플로: 신규(빨강) → 접수(주황) → 완료(초록)

## 8. 현재 미구현/제한 사항

| 항목 | 상태 | 설명 |
|------|------|------|
| Span Loss / Node Gain 토폴로지 표시 | **미구현** | SQL에서 해당 필드 주석 처리 상태. 데이터는 TB_ROADM_OPTICAL_PERFORMANCE_DAILY에 존재 |
| NTD 토폴로지 표시 | **미구현** | SQL에서 해당 필드 주석 처리 상태 |
| 영향회선 리스트 (PM) | **비활성** | SQL이 `WHERE 1=0` 으로 빈 결과 반환 |
| 영향회선 리스트 (MBA) | **비활성** | SQL이 `WHERE 1=0` 으로 빈 결과 반환 |
| opinion_cnt / new_list (PM) | **주석** | 매퍼에서 서브쿼리 주석 처리. Vue에서 참조하지만 데이터 없음 |
| PM WebSocket 실시간 알림 | **미확인** | MBA는 WebSocket 구현, PM은 API 폴링 방식 |
