# 영향회선 기능 현황 보고

> 작성일: 2026-04-01
> 작성자: UI 개발팀
> 대상 화면: 순단장애 감시, 광모듈 장애 예측 (예지보전)

---

## 1. 현황

순단장애 및 예지보전 화면의 **영향회선 리스트**가 표시되지 않는 원인을 UI 측에서 확인하였습니다.
현재 보유한 데이터로는 **UI 측 단독 개발이 불가능**하며, **엔진 측 개발이 선행**되어야 합니다.

---

## 2. 불가 사유

### 2.1 원본 테이블 미존재

영향회선 쿼리(`SELECT_MBA_PMM_INFLUENCECIRCUIT_LIST`)가 참조하는 테이블이 DB에 존재하지 않습니다.

| 항목 | 상태 |
|------|------|
| 참조 테이블 | `mba.tb_ticket_ic_mst` |
| DB 존재 여부 | **미존재** (전체 스키마 검색 결과 0건) |
| 쿼리 상태 | `WHERE 1 = 0`으로 의도적 비활성화 (빈 결과 강제 반환) |

이 테이블은 구 시스템(RCA)에서 사용하던 영향회선 테이블을 NIA 환경으로 이식할 때 **마이그레이션되지 않은 것**으로 확인됩니다.

### 2.2 대체 가능한 유사 테이블 부재

NIA DB 내에서 영향회선 역할을 할 수 있는 후보 테이블을 조사하였으나, 직접 활용이 불가합니다.

| 테이블 | 데이터 | 전표 연결 | 활용 가능 여부 |
|--------|:------:|:---------:|:--------------:|
| `nia.tb_tsdn_service_mst` | 23건 (E-Line 서비스) | 전표↔서비스 매핑 키 없음 | **불가** |
| `nia.tb_tsdn_service_node_mst` | 0건 | - | **불가** (데이터 없음) |
| `nia.tb_tsdn_service_pce_mst` | 26건 (경유 경로) | ROADM 일부만 존재 | **불가** (부분 데이터) |
| `rca.tb_rca_ticket_cable_mst` | 50,971건 | RCA 전표 전용 | **불가** (MBA 전표와 무관) |

### 2.3 노드명 불일치

서비스 마스터와 전표 시스템 간 노드 식별자가 상이하여 자동 매핑이 불가합니다.

| 시스템 | 노드 식별 | 예시 |
|--------|----------|------|
| 전표 (MBA 엔진) | 한글 장비명 | `서울NIA`, `수원성균관대` |
| 서비스 마스터 (T-SDN) | 영문 약어 | `SEOUL`, `SUWON` |

---

## 3. 영향회선 개념 설명

참고: 영향회선은 근원장애(Root Cause)와 관련되지만 다른 개념입니다.

| 개념 | 방향 | 질문 | 예시 |
|------|------|------|------|
| **근원장애** | 원인 → 결과 | "장애의 원인이 무엇인가?" | 서울NIA↔수원성균관대 구간 광레벨 3dB 이상 저하 |
| **영향회선** | 결과 → 범위 | "이 장애로 어떤 서비스가 영향받는가?" | 해당 광케이블을 경유하는 E-Line 서비스 N건 |

영향회선 기능이 동작하려면, 전표 발생 시 **해당 장애 구간을 경유하는 서비스 회선 목록**을 산출하여 저장하는 로직이 필요합니다.

---

## 4. 엔진 측 필요 개발 사항

### 4.1 영향회선 매핑 테이블 생성

전표 생성 시점에 영향받는 서비스 회선을 저장할 테이블이 필요합니다.

```
MBA.TB_MBA_TICKET_IC_MST (안)

TICKET_ID        ← FK: TB_MBA_TICKET_CURRENT
SERVICE_ID       ← 서비스 식별자
SERVICE_NAME     ← 서비스명
SRC_NODE_ID      ← 출발 노드
DST_NODE_ID      ← 도착 노드
SERVICE_TYPE     ← 서비스 유형 (E-Line 등)
INSERT_TIME      ← 생성 시각
```

### 4.2 전표 생성 시 영향회선 산출 로직

`TicketCreateService.createMbaTicket()` 실행 시:

1. 전표의 `trunk_name` + `direction`으로 장애 구간 특정
2. 해당 구간을 경유하는 서비스 회선을 `tb_tsdn_service_mst` + `tb_tsdn_service_pce_mst`에서 조회
3. 결과를 `TB_MBA_TICKET_IC_MST`에 저장

### 4.3 노드명 매핑 테이블

한글 장비명 ↔ 영문 약어 매핑이 필요합니다.

| 한글 (전표) | 영문 (서비스) |
|------------|-------------|
| 서울NIA | SEOUL |
| 수원성균관대 | SUWON |
| 대전ETRI | DAEJEON |
| 판교NOC | PANGYO |

이 매핑은 별도 테이블로 관리하거나, 기존 `nia.tb_equip_mst` 등에 컬럼 추가로 해결할 수 있습니다.

---

## 5. UI 측 대응 계획

엔진 측에서 위 개발이 완료되어 `TB_MBA_TICKET_IC_MST` 테이블에 데이터가 적재되면, UI 측에서는 아래 작업만으로 영향회선 리스트를 활성화할 수 있습니다.

1. `SELECT_MBA_PMM_INFLUENCECIRCUIT_LIST` 쿼리의 `WHERE 1 = 0` 제거
2. 주석 처리된 원본 쿼리를 새 테이블명(`TB_MBA_TICKET_IC_MST`)으로 수정
3. Vue 컴포넌트 수정 불필요 (기존 컬럼 매핑 그대로 사용 가능)

---

## 6. 요약

| 항목 | 상태 |
|------|------|
| UI 쿼리 | 비활성 (`WHERE 1 = 0`) — 정상 동작 대기 상태 |
| DB 테이블 | `mba.tb_ticket_ic_mst` 미존재 |
| 유사 데이터 | `tb_tsdn_service_mst` 23건 존재, 그러나 전표 연결 불가 |
| **UI 측 단독 해결** | **불가** |
| **필요 조치** | 엔진 측에서 영향회선 산출 로직 + 매핑 테이블 개발 |
