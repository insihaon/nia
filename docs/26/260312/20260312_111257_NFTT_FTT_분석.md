## 1. NFTT (Node Factor Ticket) — 장비 과부하 티켓

### 1.1 정의

- 상수: `RcaCodeInfo.java:244` → `TICKET_TYPE_NODE_FACTOR_TICKET = "NFTT"`
- 의미: AI가 예측한 장비(SDN 노드)의 CPU/메모리 과부하 상태를 티켓으로 발행
- 대상 테이블: `rca.tb_rca_ticket_current`

### 1.2 모듈 간 Flow


### 1.3 NFTT 생성 조건

| 항목 | 내용 |
|------|------|
| **트리거** | AI 모델이 SDN 노드의 CPU/메모리 과부하를 예측 |
| **데이터** | `NodeFactorVo` — sdn_node_id, cpu_predicted, mem_predicted, measured_datetime |
| **임계값 판단** | NiaTrafficPreprocessorApplication에서 수행 (AI 예측 결과 기반 필터링) |
| **NiaEngineApplication 역할** | 필터링된 데이터 수신 → 티켓 발행 (별도 임계값 체크 없음) |
| **대상 장비** | SDN 노드 (IP_SDN.TB_NODE에 등록된 네트워크 장비) |

### 1.4 관련 테이블

| 테이블 | 스키마 | 용도 |
|--------|--------|------|
| TB_NODEFACTOR | IP_SDN | 원본 노드팩터 (CPU_USAGE, MEM_USAGE, NODE_ID) |
| TB_NODEFACTOR | RCA | 처리된 노드팩터 (TICKET_ID 매핑) |
| TB_NODEFACTOR_AI_RESULT | RCA | AI 예측 결과 (CPU_PREDICTED, MEM_PREDICTED) |
| tb_rca_ticket_current | RCA | NFTT 티켓 저장 |
| RCA_TICKET_AL | RCA | 관련 알람 |

### 1.5 수년간 데이터 미유입 원인 추정

- NFTT insert 로직 자체는 **활성 상태** (주석 처리 아님)
- AI 모델이 과부하를 예측하는 데이터가 유입되지 않거나, 임계값을 넘는 장비가 없었던 것으로 추정
- NiaTrafficPreprocessorApplication의 필터링 단계에서 걸러져 NiaEngineApplication까지 도달하지 않은 가능성

## 2. FTT (False Traffic Ticket) — 비장애 티켓

### 2.1 정의

- 상수: **미정의** (RcaCodeInfo.java에 FTT 상수 없음, 코드 내 문자열 리터럴로만 존재)
- 의미: AI가 기존 이상트래픽/유해트래픽 티켓(ATT2, NTT)을 "비장애"로 판정한 경우
- 현재 상태: **2024-03-18에 비활성화됨**

### 2.3 과거 FTT 동작 흐름 (비활성화 전 추정)

```
이상트래픽(ATT2) 또는 유해트래픽(NTT) 티켓 발행
  ▼
AI 모델 분석 결과 수신
  ▼
Queue → NiaAiAnoToEngineMsgListener (ATT2) 또는 NiaAiNoxToEngineMsgListener (NTT)
  ▼
AI 판정: zero1_model = 1 (비장애)
  ▼
[과거 활성 로직 — 현재 전부 주석 처리]
  │
  ├─ 1) updateAiFTTResult()
  │     rca.tb_rca_ticket_current
  │     SET ticket_type = 'FTT', status = 'AUTO_FIN', root_cause_type = 'NoIssue'
  │     → 티켓을 FTT로 변환하고 자동 마감
  │
  ├─ 2) updateAutoProcessFTT()
  │     nia.tb_auto_treat_process_dtl
  │     SET self_process_type = 'F', self_process_group = 'ST', ticket_type = 'FTT'
  │     → 자가최적화 이력에 비장애 기록
  │
  ├─ 3) insertAutoProcessSop()
  │     nia.tb_sop INSERT
  │     fault_classify = '비장애', fault_type = '장애무효', status = 'AUTO_FIN'
  │     → SOP 이력 자동 생성
  │
  ├─ 4) updateFTTSop()
  │     nia.tb_sop UPDATE
  │     fault_classify = '비장애', fault_type = '장애무효'
  │     zero1_model, zero1_entropy 추가 기록
  │     → SOP에 AI 판정 데이터 보강
  │
  └─ 5) AttFttsendMail() / NttFttsendMail()
        → 비장애 판정 이메일 알림 발송
```

### 2.4 현재 동작 (비활성화 후)

```
ATT2/NTT 티켓 → AI 판정: zero1_model = 1 (비장애)
  ▼
updateAiFTTResult() [축소된 버전]
  rca.tb_rca_ticket_current
  SET zero1_model = #{model}, zero1_entropy = #{entropy}
  WHERE ticket_id = #{ticketId}
  → AI 수치만 기록. ticket_type은 ATT2/NTT 그대로 유지

※ 자동 마감(AUTO_FIN) 없음
※ SOP 자동 생성 없음
※ 이메일 알림 없음
※ 티켓은 수동 처리 대기 상태로 남음
```

### 2.5 비활성화 시점 및 근거

- **시점**: 2024년 3월 18일 (`AutoProcessMapper.xml:281` 주석: `<!--240318 수정 , FTT 원복 시 아래 쿼리 사용-->`)
- **방식**: 코드 삭제가 아닌 주석 처리 → "FTT 원복 시" 복원 의도 명시
- **설정 플래그 없음**: application.yml 등에 on/off 설정 없이 코드 레벨에서 직접 비활성화

### 2.6 비활성화 원인 추정

1. **AI 오판 가능성**: AI의 비장애 판정(zero1_model=1)이 부정확하여, 실제 장애를 자동 마감(AUTO_FIN)시키는 사고 발생 우려
2. **자동화 범위 축소**: 티켓 자동 마감은 리스크가 크므로, AI 수치만 기록하고 최종 판단은 운영자에게 위임하는 방향으로 전환
3. **복원 가능성 열어둠**: 주석에 "FTT 원복 시 아래 쿼리 사용"이라 명시 → AI 정확도가 검증되면 재활성화 의도

## 3. 비교 요약

| 구분 | NFTT | FTT |
|------|------|-----|
| **정식 명칭** | Node Factor Ticket (장비 과부하) | False Traffic Ticket (비장애) |
| **상수 정의** | `RcaCodeInfo.TICKET_TYPE_NODE_FACTOR_TICKET` | 없음 (문자열 리터럴) |
| **INSERT 로직** | 활성 | 없음 (INSERT 자체가 아닌 UPDATE로 기존 티켓 변환) |
| **코드 상태** | 정상 활성 | 2024-03-18 주석 처리 |
| **데이터 유입** | 수년간 없음 (조건 미충족) | 불가능 (코드 비활성화) |
| **복원 가능성** | 해당 없음 (이미 활성) | 주석 해제 시 즉시 가능 |
