# 자가 처리 현황 산정기준

> 대시보드 > 자가 처리 현황 차트의 4개 항목에 대한 산정기준 정리

---

## 원본 테이블

`nia.tb_auto_treat_process_dtl` (자동조치 프로세스 상세)

## 핵심 컬럼

| 컬럼 | 값 | 의미 |
|------|---|------|
| `self_process_group` | `'SO'` / `'ST'` | SO=자가최적화, ST=자가회복 |
| `clos_type` | `'A'` 등 | A=자동 마감 |
| `clos_status` | `'Y'` / `'N'` | Y=마감 완료, N=미완료 |
| `occur_time` | timestamp | 발생 시각 |

---

## 4개 항목 산정기준

| 차트 항목 | SQL 필드 | 조건 | 의미 |
|----------|----------|------|------|
| 자가최적화 총 발생 | `so_totalcount` | `self_process_group = 'SO'` | SO 그룹의 전체 발생 건수 (성공/실패 무관) |
| 자가최적화 건 수 | `so_count` | `self_process_group = 'SO'` AND `clos_type = 'A'` AND `clos_status = 'Y'` | SO 그룹 중 자동 마감 성공 건수 |
| 자가회복 총 발생 | `st_totalcount` | `self_process_group = 'ST'` | ST 그룹의 전체 발생 건수 (성공/실패 무관) |
| 자가회복 건 수 | `st_count` | `self_process_group = 'ST'` AND `clos_type = 'A'` AND `clos_status = 'Y'` | ST 그룹 중 자동 마감 성공 건수 |

### 해석

- **총 발생** = 자동조치가 시도된 전체 건수
- **건 수** = 그 중 자동 마감(`clos_type='A'`)이 성공(`clos_status='Y'`)한 건수
- 두 값의 차이가 클수록 자동 처리 실패율이 높음

## SQL 쿼리

**위치**: `apps/Web2024/BE/app-nia/src/main/resources/mapper/db2nd/NiaMapper.xml`
**SQL ID**: `SELECT_SELF_PROCESS_STATISTICS`

```sql
WITH
series AS (
    SELECT generate_series (
        DATE_TRUNC(#{SERIES_TYPE}, TO_DATE(#{DATE}, #{SERIES_FORMAT})::date),
        DATE_TRUNC(#{SERIES_TYPE}, TO_DATE(#{DATE}, #{SERIES_FORMAT})::date)
            + ('1 ' || #{SERIES_TYPE})::INTERVAL
            - ('1 ' || #{STATISTICS_TYPE})::INTERVAL,
        ('1 '|| #{STATISTICS_TYPE})::INTERVAL
    ) AS series_time
)
SELECT
    TO_CHAR(series.series_time, #{STATISTICS_FORMAT}) AS series_time,
    COALESCE(COUNT(CASE WHEN TPD.self_process_group IN ('SO') THEN 1 END), 0)
        AS SO_TOTALCOUNT,
    COALESCE(COUNT(CASE WHEN TPD.self_process_group IN ('SO')
        AND TPD.clos_type = 'A' AND TPD.clos_status = 'Y' THEN 1 END), 0)
        AS SO_COUNT,
    COALESCE(COUNT(CASE WHEN TPD.self_process_group IN ('ST') THEN 1 END), 0)
        AS ST_TOTALCOUNT,
    COALESCE(COUNT(CASE WHEN TPD.self_process_group IN ('ST')
        AND TPD.clos_type = 'A' AND TPD.clos_status = 'Y' THEN 1 END), 0)
        AS ST_COUNT
FROM series
LEFT JOIN nia.tb_auto_treat_process_dtl TPD
    ON DATE_TRUNC(#{STATISTICS_TYPE}, TPD.occur_time) = series.series_time
GROUP BY series_time
ORDER BY series_time
```

### SO (자가최적화) 프로세스 상세

**목적**: 이상/유해 트래픽 감지 시 자동으로 포트를 차단하여 네트워크 최적화

**레코드 생성 시점 (tb_auto_treat_process_dtl)**:

| 생성 시점 | self_process_type | 트리거 |
|----------|-------------------|--------|
| 이상트래픽 티켓 생성 | `A` (ATT2) | 엔진 프로세서가 트래픽 이상 감지 |
| 유해트래픽 AI 감지 | `N` (NTT) | AI 모델이 유해 트래픽 판정 |
| 유해트래픽 일반 감지 | `N` (NTT) | 일반 유해 트래픽 탐지 |

**자동 마감 흐름** (`TicketClearPassing5Minute`):
```
티켓 생성 → 5분 경과 → 스케줄러 감지
→ tb_rca_ticket_current 상태를 AUTO_FIN으로 변경
→ tb_sop 레코드 생성 (autoprocType = 'SOA')
→ tb_auto_treat_process_dtl 레코드 생성 (group='SO')
→ IP-SDN API 호출하여 포트 차단 (shutdown)
→ clos_status = 'Y', clos_time = now()
```

### ST (자가회복) 프로세스 상세

**목적**: Syslog 알람(IF_UP/IF_DOWN 등) 감지 시 자동으로 포트 상태를 복구

**레코드 생성 시점**:

| 생성 시점 | self_process_type | 트리거 |
|----------|-------------------|--------|
| Syslog 알람 5분 경과 | `S` (Syslog) | 스케줄러가 5분 초과 알람 감지 |
| AI 오탐 판정 (FTT 전환) | `F` (FTT) | AI model=1 (정상) 판정 시 SO→ST로 전환 |

**자동 마감 흐름** (`SyslogClearPassing5MinuteAndRemoteCommand`):
```
Syslog 알람 수신 → 5분 경과 → 스케줄러 감지
→ tb_syslog_alarm_mst 상태를 AUTO_FIN으로 변경
→ tb_syslog_sop 레코드 생성
→ tb_auto_treat_process_dtl 레코드 생성 (group='ST', type='S')
→ IP-SDN API 호출하여 포트 복구 (shutdown → noshut)
  (현재 포트 제어 코드는 주석 처리 상태)
```

### FTT 전환 (SO → ST)

AI가 오탐(false alarm)으로 재판정하면, 기존 SO 레코드가 ST로 전환됩니다:

```sql
UPDATE tb_auto_treat_process_dtl
SET self_process_type = 'F',
    self_process_group = 'ST',
    ticket_type = 'FTT'
WHERE ticket_id = #{ticketId}
```

### IP-SDN 포트 제어 상세 (SelfProcessService)

> 소스: `SelfProcessService.java`, `EngineClearSchedulerJobServiceImpl.java`

#### API 엔드포인트

| 동작 | IP-SDN 엔드포인트 | 파라미터 |
|------|------------------|---------|
| 포트 차단 | `/ipsdn/services/config/interfaces/shutdown` | `nodename`, `ifname` |
| 포트 복구 | `/ipsdn/services/config/interfaces/noshut` | `nodename`, `ifname` |

#### 코드 활성화 현황

| 코드 경로 | 코드 상태 | 실제 동작 | 비고 |
|----------|----------|----------|------|
| `requestNiaApi()` | **활성** | HTTP POST 실행 | serviceUrl이 비어있으면 무의미 |
| `UserTicketFeedbackCheck()` | **활성** | closType='A'일 때 호출 | requestNiaApi 호출 |
| `UserSyslogFeedbackCheck()` | **활성** | closType='A'일 때 호출 | requestNiaApi 호출 |
| `TicketPortDownCommand()` | **비활성** | serviceUrl = `""` (빈 문자열) | 코드는 활성이나 URL이 비어 실행 무효 |
| `SyslogPortDownCommand()` | **비활성** | shutdownUrl, noshutUrl = `""` | 코드는 활성이나 URL이 비어 실행 무효 |
| `SyslogClearPassing5MinuteAndRemoteCommand()` 내 포트 제어 | **주석 처리** | 실행 안됨 | 의도적으로 비활성화 |
| 스케줄러 `TicketPortDownCommand` 직접 호출 | **주석 처리** | 실행 안됨 | SchdulerServiceImpl.java 45-48행 |
| 스케줄러 `SyslogPortDownCommand` 직접 호출 | **주석 처리** | 실행 안됨 | SchdulerServiceImpl.java 50-53행 |

#### 결론

**포트 제어 기능은 아키텍처적으로 완성되어 있으나, 운영상 비활성화 상태:**

1. `requestNiaApi()` → `UserTicketFeedbackCheck()` → `TicketPortDownCommand()` 호출 체인은 활성 코드
2. 그러나 `TicketPortDownCommand()`에서 `serviceUrl = ""`로 세팅하여 **실제 API 호출이 발생하지 않음**
3. Syslog 쪽은 포트 제어 호출 자체가 **주석 처리**
4. 주석에 남아있는 참조 URL: `http://203.255.249.31:8088/ipsdn/services/config/interfaces/shutdown?nodename=daejeon-5812&ifname=xe13`
5. 포트 제어를 활성화하려면 `serviceUrl`에 실제 IP-SDN API URL을 설정해야 함

## "총 발생"과 "건 수" 차이 원인 분석

> 대시보드에서 총 발생 건수와 건 수가 대부분 다른 이유

### clos_status 생명주기

```
INSERT (레코드 생성)
  → clos_status = 'N', clos_time = null

UPDATE (마감 처리)
  → clos_status = 'Y', clos_time = now()
```

- **총 발생**: `clos_status`와 무관하게 전체 COUNT
- **건 수**: `clos_status = 'Y'` AND `clos_type = 'A'`인 것만 COUNT
- 따라서 `clos_status = 'N'`으로 남은 레코드가 많을수록 두 값의 차이가 커짐

### clos_status가 'N'으로 남는 원인

#### 원인 1 (주요): ST(자가회복) 마감 코드가 주석 처리

`SyslogClearPassing5MinuteAndRemoteCommand()` 내부에서 `UserSyslogFeedbackCheck()` 호출이 주석 처리되어 있어, **모든 ST 레코드는 영원히 clos_status = 'N'으로 남음.**

```java
// EngineClearSchedulerJobServiceImpl.java 328-333행
// selfProcessService.UserSyslogFeedbackCheck(shutdownUrl, alarmno);  ← 주석!
```

이로 인해 차트에서 **"자가회복 건 수"는 항상 0에 가까움.**

#### 원인 2: SO(자가최적화) 티켓이 AUTO_FIN/FIN에 미도달

`TicketPortDownCommand()`가 마감 대상을 조회하는 SQL:

```sql
SELECT ... FROM nia.tb_auto_treat_process_dtl auto
  JOIN rca.tb_rca_ticket_current rca ON ...
WHERE auto.clos_status = 'N'
  AND (rca.status = 'AUTO_FIN' OR rca.status = 'FIN')
```

티켓이 INIT, ACK 등 중간 상태에 멈추면 이 쿼리에 잡히지 않아 `clos_status = 'N'` 유지.

#### 원인 3: selectAutoProcess null 반환 시 NPE

`UserTicketFeedbackCheck()` 진입 직후:

```java
AutoTreatProcessVo autoTreatProcessVo = autoProcessMapper.selectAutoProcess(num);
// autoTreatProcessVo가 null이면 아래에서 NPE 발생
if ("A".equals(autoTreatProcessVo.getClosType())) { ... }  // NPE!
// → catch 블록으로 이동 → updateAutoProcessTicket() 실행 안됨 → 'N' 유지
```

#### 참고: requestNiaApi("") 빈 URL은 영향 없음

`requestNiaApi("")` 호출 시 `new URL("")`에서 MalformedURLException이 발생하지만, **메서드 내부 try-catch에서 삼켜서 예외가 전파되지 않음.** 따라서 빈 URL 자체는 `clos_status` 업데이트를 막지 않음.

### 원인별 영향도

| 원인 | 대상 | 영향도 | 결과 |
|------|------|--------|------|
| ST 마감 코드 주석 처리 | 자가회복 전체 | **매우 높음** | 자가회복 건 수 ≈ 0 |
| 티켓 AUTO_FIN 미도달 | 자가최적화 일부 | 중간 | SO 총 발생 > SO 건 수 |
| selectAutoProcess null | 자가최적화 소수 | 낮음 | 간헐적 미마감 |

### 차트 예상 패턴

```
자가최적화: 총 발생 > 건 수  (일부 차이)
자가회복:   총 발생 >> 건 수  (건 수가 거의 0)
```
