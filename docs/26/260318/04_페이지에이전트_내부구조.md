# 1. Page-Agent 동작 원리

## 1.1 한 줄 요약

**Page-Agent = DOM 추출기(PageController) + LLM 에이전트 루프(Core) + UI 패널(Panel)**

페이지의 인터랙티브 요소를 텍스트로 변환 → LLM에게 "뭘 클릭/입력할지" 물어봄 → LLM 응답대로 DOM 조작 실행 → 다시 DOM 추출 → 반복

## 1.2 전체 흐름도 (Re-Act Agent Loop)

```
사용자: "SOP 이력에서 '광케이블' 키워드로 검색해줘"
                │
                ▼
┌─────────────────────────────────────────────────────┐
│  PageAgent.execute(task)                             │
│                                                      │
│  while (step < maxSteps) {                          │
│                                                      │
│    ┌──────────────────────────────────┐              │
│    │ 1. OBSERVE (관찰)                │              │
│    │    PageController.getBrowserState()│             │
│    │    → DOM을 텍스트로 변환          │              │
│    │    → 인터랙티브 요소에 번호 부여  │              │
│    └──────────┬───────────────────────┘              │
│               │                                      │
│               ▼                                      │
│    ┌──────────────────────────────────┐              │
│    │ 2. THINK (사고)                  │              │
│    │    LLM API 호출                  │              │
│    │    system prompt + agent history │              │
│    │    + browser state 전송          │              │
│    │    → LLM이 다음 행동 결정        │              │
│    └──────────┬───────────────────────┘              │
│               │                                      │
│               ▼                                      │
│    ┌──────────────────────────────────┐              │
│    │ 3. ACT (실행)                    │              │
│    │    LLM 응답 파싱                 │              │
│    │    → 도구 실행 (click, input 등) │              │
│    │    → 결과를 history에 기록       │              │
│    └──────────┬───────────────────────┘              │
│               │                                      │
│               ▼                                      │
│    action이 "done"이면 종료, 아니면 다시 1로         │
│  }                                                   │
└─────────────────────────────────────────────────────┘
```

## 1.3 OBSERVE 단계 상세: DOM을 LLM이 읽을 수 있는 텍스트로 변환

이것이 page-agent의 핵심 기술입니다. 스크린샷이 아닌 **텍스트 기반** DOM 추출을 사용합니다.

### DOM 추출 과정

```
실제 DOM
├── <nav>
│   ├── <button>SOP 이력</button>
│   ├── <button>장애이력</button>
│   └── <input placeholder="검색">
├── <table>
│   └── <tr><td>데이터...</td></tr>
└── <button>조회</button>

          │ PageController.updateTree()
          │ (getFlatTree → flatTreeToString)
          ▼

LLM이 받는 텍스트 (simplified HTML)
─────────────────────────────
[0]<button>SOP 이력</button>
[1]<button>장애이력</button>
[2]<input placeholder="검색" />
데이터...
[3]<button>조회</button>
```

**핵심 로직** (`getFlatTree` → `flatTreeToString`):
1. 전체 DOM 트리를 순회하면서 **가시적(visible)** 요소만 필터
2. 인터랙티브 요소(button, input, a, select 등)에 **번호 인덱스** `[0]`, `[1]`, `[2]` 부여
3. 각 요소의 `ref` (실제 HTMLElement 참조)를 `selectorMap`에 저장
4. 비인터랙티브 텍스트 노드는 인덱스 없이 그대로 포함
5. 부모-자식 관계를 들여쓰기(탭)로 표현
6. **새로 나타난 요소**는 `*[N]` 형태로 마킹 (LLM이 변화를 인식)

### BrowserState 구조

```typescript
interface BrowserState {
  url: string;      // 현재 페이지 URL
  title: string;    // 페이지 타이틀
  header: string;   // "Page info: 1920x1080px...\n[Start of page]"
  content: string;  // 위의 simplified HTML (인터랙티브 요소 + 텍스트)
  footer: string;   // "... 300 pixels below ..." 또는 "[End of page]"
}
```

### viewportExpansion

기본적으로 뷰포트에 보이는 요소만 추출하지만, `viewportExpansion` 설정으로 뷰포트 바깥의 요소도 포함 가능. 이를 통해 스크롤 없이도 아래쪽 요소를 인식할 수 있음.

## 1.4 THINK 단계 상세: LLM API 호출

### LLM에게 보내는 메시지 구조

```
[system message]
  ├── SYSTEM_PROMPT (고정, ~3000 tokens)
  │   ├── 역할 정의: "You are an AI agent designed to operate in an iterative loop"
  │   ├── 입력 형식 설명: agent_history, agent_state, browser_state
  │   ├── browser_rules: 인덱스 사용법, 스크롤 규칙, 캡챠 대응 등
  │   ├── reasoning_rules: 이전 액션 평가, 메모리 활용, 진행 상황 분석
  │   └── task_completion_rules: done 액션 호출 조건
  │
[user message]
  ├── <instructions> (선택)
  │   ├── <system_instructions>: 개발자가 설정한 도메인 지식
  │   ├── <page_instructions>: 현재 페이지 맞춤 지침
  │   └── <llms_txt>: 사이트의 llms.txt (실험적)
  │
  ├── <agent_state>
  │   ├── <user_request>: 사용자가 입력한 태스크
  │   └── <step_info>: "Step 3 of 40 max possible steps"
  │
  ├── <agent_history>
  │   ├── <step_1>: 이전 스텝의 평가/메모리/목표/결과
  │   ├── <step_2>: ...
  │   └── <sys>: 시스템 관찰 메시지 (네비게이션, 경고 등)
  │
  └── <browser_state>
      ├── header: 페이지 정보
      ├── content: simplified HTML (인터랙티브 요소 목록)
      └── footer: 스크롤 정보
```

### MacroTool 패턴 (핵심 설계)

page-agent는 LLM의 tool_call API를 사용하되, **모든 도구를 하나의 MacroTool(AgentOutput)**로 묶어서 호출합니다.

```json
// LLM에게 제공되는 tool 스키마 (간소화)
{
  "name": "AgentOutput",
  "parameters": {
    "evaluation_previous_goal": "string (이전 행동 평가)",
    "memory": "string (기억할 정보)",
    "next_goal": "string (다음 목표)",
    "action": {
      // 아래 중 하나를 선택:
      "click_element_by_index": { "index": "number" },
      "input_text": { "index": "number", "text": "string" },
      "select_dropdown_option": { "index": "number", "text": "string" },
      "scroll": { "down": "boolean", "num_pages": "number" },
      "done": { "text": "string", "success": "boolean" },
      "wait": { "seconds": "number" },
      "ask_user": { "question": "string" },
      // + customTools로 추가한 도구들
    }
  }
}
```

**왜 MacroTool인가?**
- LLM이 한 번의 응답에서 **사고(reflection) + 행동(action)**을 함께 출력
- 별도의 "reasoning" 호출이 불필요 → 토큰 절약
- tool_call이 아닌 text로 응답해도 JSON을 파싱하여 복구 (`normalizeResponse`)

### LLM 응답 복구 (normalizeResponse)

LLM이 깔끔한 tool_call을 반환하지 않을 때를 대비해 5단계 복구 로직이 있음:

```
#1: tool_call은 있지만 name이 AgentOutput이 아닌 경우 → action으로 감싸기
#2: tool_call 없이 content에 JSON → 파싱
#3: content JSON이 function 형태 → 풀어서 파싱
#4: action 키가 없는 JSON → action으로 감싸기
#5: 모든 복구 실패 → wait 1초 기본 액션
```

### 모델별 패치 (modelPatch)

LLM 종류에 따라 요청 body를 자동 조정:

| 모델 | 패치 내용 |
|------|----------|
| Qwen | temperature 최소 1.0, thinking 비활성화 |
| Claude | thinking 비활성화, tool_choice 포맷 변환 |
| Gemini | reasoning_effort: minimal |
| GPT-5/5-mini | reasoning_effort: low/none |
| Grok | tool_choice 제거, reasoning 비활성화 |

**NIA 프로젝트는 Gemini 2.5-flash를 사용** → reasoning_effort: minimal 패치 적용됨

## 1.5 ACT 단계 상세: DOM 조작 실행

### 내장 도구 (Tools)

| 도구 | 설명 | DOM 조작 방식 |
|------|------|-------------|
| `click_element_by_index` | 요소 클릭 | scrollIntoView → movePointer → mousedown/focus/mouseup/click 이벤트 시뮬레이션 |
| `input_text` | 텍스트 입력 | 클릭 → nativeInputValueSetter로 값 설정 → input/change 이벤트 발화 |
| `select_dropdown_option` | 드롭다운 선택 | 클릭 → 옵션 텍스트 매칭 → 옵션 클릭 |
| `scroll` / `scroll_horizontally` | 스크롤 | window.scrollBy 또는 element.scrollBy |
| `wait` | 대기 | setTimeout |
| `done` | 완료 | 태스크 종료, 결과 반환 |
| `ask_user` | 사용자에게 질문 | onAskUser 콜백 호출 |
| `execute_javascript` | JS 실행 (실험적) | eval (experimentalScriptExecutionTool: true 시) |

### 클릭 시뮬레이션 상세

```javascript
async function clickElement(element) {
  blurLastClickedElement();          // 이전 포커스 해제
  scrollIntoViewIfNeeded(element);   // 뷰포트로 스크롤
  movePointerToElement(element);     // 포인터 애니메이션 (UI용)
  // 브라우저 이벤트 순서 그대로 시뮬레이션:
  element.dispatchEvent(new MouseEvent("mouseenter", ...));
  element.dispatchEvent(new MouseEvent("mouseover", ...));
  element.dispatchEvent(new MouseEvent("mousedown", ...));
  element.focus();
  element.dispatchEvent(new MouseEvent("mouseup", ...));
  element.dispatchEvent(new MouseEvent("click", ...));
}
```

### 텍스트 입력 시뮬레이션

```javascript
// React/Vue 등 프레임워크가 value 변경을 감지하도록
// 네이티브 setter를 사용하여 input.value 설정
const nativeInputValueSetter = Object.getOwnPropertyDescriptor(
  HTMLInputElement.prototype, "value"
).set;

nativeInputValueSetter.call(element, text);
element.dispatchEvent(new Event("input", { bubbles: true }));
element.dispatchEvent(new Event("change", { bubbles: true }));
```

## 1.6 에이전트 루프 제어

### 종료 조건
- LLM이 `done` 액션 호출 시
- `maxSteps` (기본 40) 초과 시
- 에러 발생 시 (네트워크, 인증 등)
- `agent.stop()` 호출 시 (AbortController)

### 관찰(Observation) 시스템
매 스텝마다 자동으로 주입되는 시스템 메시지:
- 누적 대기 시간 3초 초과: "더 이상 기다리지 마세요"
- 페이지 URL 변경: "Page navigated to → ..."
- 남은 스텝 5개: "마무리하거나 done을 호출하세요"
- 남은 스텝 2개: "즉시 done을 호출하세요!"

### 리트라이 메커니즘
LLM 호출 실패 시 자동 재시도:
- 재시도 가능: 네트워크 오류, 429 (Rate Limit), 5xx, 도구 인자 오류
- 재시도 불가: 인증 오류 (401/403), 컨텐츠 필터, context length 초과

# 2. 모듈 구조 및 소스코드 분석

## 2.1 패키지 구조

```
page-agent (25줄, 진입점)
├── @page-agent/core (826줄) — 에이전트 루프, 시스템 프롬프트, 도구 정의
│   ├── @page-agent/llms (406줄) — LLM API 호출, 모델 패치, 에러 처리
│   └── @page-agent/page-controller (2012줄) — DOM 추출, 요소 조작, 이벤트 시뮬레이션
└── @page-agent/ui (804줄) — 채팅 패널 UI, 이벤트 표시

총 코드: ~4,073줄 (minified ESM 기준)
디스크 크기: page-agent 303KB + @page-agent 1.4MB = ~1.7MB
외부 의존성: chalk, zod/v4, ai-motion
```

## 2.2 모듈별 역할

### page-agent (진입점, 25줄)
```javascript
class PageAgent extends PageAgentCore {
  panel;  // UI 패널
  constructor(config) {
    const pageController = new PageController({ ...config, enableMask: true });
    super({ ...config, pageController });
    this.panel = new Panel(this, { language: config.language });
  }
}
```
단순히 Core + PageController + UI Panel을 조합하는 역할.

### @page-agent/core (826줄) — 두뇌
- `SYSTEM_PROMPT`: 약 3000토큰의 고정 시스템 프롬프트
- `tools`: 내장 도구 8개 정의 (done, wait, ask_user, click, input_text, select, scroll, scroll_h, execute_js)
- `PageAgentCore.execute()`: 메인 에이전트 루프
- `normalizeResponse()`: LLM 응답 복구 로직
- `packMacroTool()`: 모든 도구를 하나의 MacroTool로 합성
- `assembleUserPrompt()`: LLM에게 보낼 user message 조립
- `getInstructions()`: system/page instructions + llms.txt 로딩

### @page-agent/page-controller (2012줄) — 눈과 손
- `getFlatTree()`: DOM 트리 → 평면 트리 변환
- `flatTreeToString()`: 평면 트리 → LLM용 텍스트 변환
- `clickElement()`: 클릭 시뮬레이션 (6단계 이벤트)
- `inputTextElement()`: 텍스트 입력 (네이티브 setter + 이벤트)
- `selectOption()`: 드롭다운 옵션 선택
- `scrollIntoViewIfNeeded()`: 자동 스크롤
- `SimulatorMask`: 자동화 중 사용자 입력 차단 오버레이

### @page-agent/llms (406줄) — 통신
- `OpenAIClient`: OpenAI-compatible API 호출 (fetch 기반)
- `zodToOpenAITool()`: Zod 스키마 → OpenAI function calling 스키마 변환
- `modelPatch()`: 모델별 요청 body 패치
- `InvokeError`: 에러 분류 (retryable vs non-retryable)
- `LLM` 클래스: 리트라이 로직, 에러 이벤트 발화

### @page-agent/ui (804줄) — 얼굴
- `Panel`: 채팅 패널 (입력창, 메시지 목록, 상태 표시)
- `show()`/`hide()`: 패널 토글
- 이벤트 리스닝: statuschange, historychange, activity

# 3. 커스텀 빌드 타당성 검토

## 3.1 "커스텀 빌드"란?

page-agent의 node_modules 의존성을 제거하고, 소스코드를 분해하여 프로젝트 내부에 직접 포함시키는 것.

## 3.2 커스텀 빌드의 잠재적 장점

| # | 장점 | 현실적 평가 |
|---|------|------------|
| 1 | DOM 추출 로직 커스터마이징 | **유의미**. el-date-picker 등 Element UI 컴포넌트를 특별 처리 가능. 예: 캘린더 팝업을 "날짜선택기(현재값: 2026-03-01)" 같은 의미있는 텍스트로 변환 |
| 2 | 시스템 프롬프트 완전 제어 | customSystemPrompt로 이미 가능. **불필요** |
| 3 | 도구 추가/제거 | customTools로 이미 가능. **불필요** |
| 4 | LLM API 호출 커스터마이징 | customFetch로 이미 가능. **불필요** |
| 5 | UI 패널 커스터마이징 | Panel을 사용 안 하고 자체 UI 연동 가능 (PageAgentCore 직접 사용). **불필요** |
| 6 | 번들 크기 감소 | 현재 1.7MB, 미미함. **불필요** |
| 7 | 이벤트 시뮬레이션 방식 변경 | **유의미**. Vue 2 컴포넌트에 맞는 이벤트 발화 순서 조정 가능 |
| 8 | 업스트림 업데이트에서 독립 | 양날의 검. 보안/버그 패치를 직접 해야 함. **대체로 단점** |

**실질적으로 유의미한 장점: #1(DOM 추출), #7(이벤트 시뮬레이션) 단 2가지**

## 3.3 커스텀 빌드의 단점

| # | 단점 | 심각도 |
|---|------|--------|
| 1 | **유지보수 부담**: page-agent 업데이트(버그픽스, 새 모델 지원)를 직접 반영해야 함 | 높음 |
| 2 | **초기 작업량**: 4000줄+ 코드를 이해하고 빌드 시스템에 통합 | 중간 |
| 3 | **테스트 부재**: 커스터마이징한 코드의 회귀 테스트를 직접 작성해야 함 | 중간 |
| 4 | **모델 패치 누락 위험**: 새 LLM 출시 시 modelPatch 업데이트 필요 | 중간 |
| 5 | **라이선스**: MIT이므로 법적 문제는 없으나, attribution 필요 | 낮음 |

## 3.4 난이도 평가

### 전체 포크 (모든 모듈을 프로젝트에 포함)

| 항목 | 난이도 | 소요 시간 |
|------|--------|----------|
| 소스 분석 및 이해 | 중간 | 2-3일 |
| 빌드 시스템 통합 (webpack/vue-cli) | 중간 | 1일 |
| chalk/zod/ai-motion 의존성 처리 | 쉬움 | 반일 |
| DOM 추출 커스터마이징 | 높음 (2012줄) | 3-5일 |
| 이벤트 시뮬레이션 수정 | 중간 | 1-2일 |
| 테스트 작성 | 높음 | 2-3일 |
| **총합** | **중간~높음** | **약 2-3주** |

### 부분 포크 (page-controller만 포크)

|------|--------|----------|
| page-controller 소스 분석 | 중간 | 1-2일 |

| 나머지는 npm 패키지 그대로 사용 | 쉬움 | 반일 |
| **총합** | **중간** | **약 1주** |

## 3.5 대안 비교: 커스텀 빌드 vs 기존 API 활용

### customTools + instructions로 해결 가능한 것 (코드 변경 없음)

```
✅ 날짜 선택 → set_date customTool
✅ 화면 이동 → navigate_to customTool
✅ 도메인 지식 → instructions.system
✅ 화면별 힌트 → instructions.getPageInstructions
✅ 알람 요약 → get_alarm_summary customTool
✅ 데이터 내보내기 → export_data customTool
✅ DOM 추출 변환 → transformPageContent 콜백
✅ LLM 요청 커스터마이징 → customFetch
```

### 커스텀 빌드로만 해결 가능한 것

```
⚠️ DOM 추출 시 Element UI 컴포넌트를 의미론적으로 인식
   → 예: el-date-picker를 "[DatePicker: 2026-03-01~2026-03-31]"로 추출
   → 단, transformPageContent로 사후 보정이 가능하므로 필수는 아님

⚠️ 클릭/입력 이벤트 시뮬레이션 방식 변경
   → 예: Vue 2의 v-model 바인딩에 최적화된 이벤트 순서
   → 단, customTools + executeJavascript로 우회 가능

⚠️ interactiveBlacklist/Whitelist 동적 제어
   → 이미 DomConfig에서 지원하지만, 런타임 동적 변경은 포크 필요
```

## 3.6 결론 및 권장

### 결론: **커스텀 빌드는 현 시점에서 불필요**

```
┌────────────────────────────────────────────────────────────┐
│                                                            │
│   page-agent의 기존 확장 API가 충분히 강력함                │
│                                                            │
│   customTools     → 날짜, 네비게이션, 데이터 추출 문제 해결 │
│   instructions    → 도메인 지식, 화면별 힌트 주입           │
│   transformPageContent → DOM 추출 결과 사후 보정            │
│   experimentalScriptExecutionTool → 최후 수단 JS 실행      │
│   customSystemPrompt → 시스템 프롬프트 완전 교체            │
│                                                            │
│   이 API들로 해결되지 않는 문제가 발견될 때 포크를 검토     │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 권장 접근 순서

```
1단계: 기존 API 최대 활용 (지금 바로 가능)
  └── customTools, instructions, transformPageContent 적용
  └── docs/page-agent-scenarios.md의 날짜 해결 Plan A 실행

2단계: 한계 도달 시 부분 확장 검토
  └── page-controller의 DomConfig 활용
  └── interactiveBlacklist/Whitelist로 불필요 요소 제외
  └── viewportExpansion으로 추출 범위 확대

3단계: 그래도 부족하면 부분 포크 (page-controller만)
  └── DOM 추출 로직만 커스터마이징
  └── 나머지 모듈은 npm 패키지 유지
  └── 예상 소요: 1주

4단계: 전체 포크 (사실상 필요 없을 가능성 높음)
  └── 모든 모듈을 프로젝트에 포함
  └── 예상 소요: 2-3주
  └── 이 단계가 필요한 경우: page-agent 프로젝트가 abandoned 되거나,
      근본적으로 다른 DOM 추출 전략이 필요한 경우
```

### "굳이 그렇게까지 할 필요가 있는가?"

**없다.** page-agent v1.5.7은 충분히 잘 설계된 확장 API를 제공하고 있으며, NIA 프로젝트에서 겪는 대부분의 문제(날짜 선택, 복합 컴포넌트)는 `customTools`로 해결 가능합니다. 커스텀 빌드는 투입 대비 효과가 낮습니다.
