# 순단장애 / 예지보전 — 관리자 매뉴얼

> 최종 갱신: 2026-04-06

---

## 1. 화면 개요

### 순단장애 감시 (TransientOutageList)

ROADM 광 전송 구간에서 발생한 **회선 순단(광레벨 급격한 저하)** 장애를 실시간으로 감시하고 처리하는 화면.

**주요 기능:**
- 장애 티켓 실시간 수신 및 목록 표시
- 2D 토폴로지 (캐리어 구간의 ROADM 노드 배치)
- 알람 리스트 (저광레벨 경보 상세)
- 인지/마감 처리
- TT 일괄처리 (동일 구간 티켓 일괄 처리)

### 예지보전 (PredictiveMaintenanceTicket)

ROADM 광모듈의 성능 데이터를 AI로 분석하여 **장기적인 장애 발생을 사전 예측**하는 화면.

**주요 기능:**
- 예측 경보 티켓 목록
- 2D 토폴로지 (구간별 광 성능 지표 표시)
- 운용자 검토 의견 등록/관리

## 2. 순단장애 — 화면 구성

### 2.1 화면 레이아웃

```
┌──────────────────────────────────────────────────────────────┐
│  상단바: 순단장애 감시 | 전체 N건 | 필터 | 알람ON/OFF        │
├──────────┬───────────────────────────────────────────────────┤
│          │  [2D 토폴로지]                                    │
│  티켓    │  캐리어 구간의 ROADM 노드 시각화                   │
│  목록    │  노드 클릭 → 해당 노드 알람 필터링                 │
│          ├───────────────────────────────────────────────────┤
│  (카드   │  [알람 리스트]                                     │
│   형태)  │  저광레벨 경보 상세 (포트, 값, 레벨)               │
│          │  Critical / Major / Minor / Warning / Clear       │
└──────────┴───────────────────────────────────────────────────┘
```

### 2.2 티켓 카드 정보

각 티켓 카드에 표시되는 정보:

| 영역 | 항목 | 설명 |
|------|------|------|
| 좌측 | 상태 배지 | INIT(대기), ACK(인지), FIN(마감), AUTO_FIN(자동마감) |
| 좌측 | 장애유형 | 순단장애 |
| 좌측 | 전체 알람 수 | 해당 티켓의 총 경보 건수 |
| 우측 | A측/Z측 시스템 | 장애 구간의 양 끝 ROADM 장비명 |
| 우측 | 캐리어명 (trunk_name) | 광 전송 구간 이름 (예: 서울NIA--수원성균관대) |
| 우측 | 방향 | UP 또는 DOWN (광 신호 방향) |
| 하단 | 최초 장애 발생 시간 | 엔진이 감지한 최초 시간 |
| 하단 | 인지/마감/TT일괄처리/Map 버튼 | 처리 액션 |

### 2.3 알람 리스트 컬럼

| 컬럼 | 설명 |
|------|------|
| 시스템명 (sysname) | ROADM 장비명 |
| 포트 (ptpname) | 광 포트 식별자 |
| IN/OUT | 광 신호 입출력 구분 |
| 장비유형 (equip_type) | ROADM 모델명 |
| 최대값 (value_max) | 측정 구간 내 최대 광레벨 (dBm) |
| 현재값 (value_cur) | 현재 광레벨 (dBm) |
| 이전값 (old_value_cur) | 15분 전 광레벨 (dBm) |
| 최소값 (value_min) | 측정 구간 내 최소 광레벨 (dBm) |
| 변동폭 (diff) | value_max - value_min |
| 이전 변동폭 (old_diff) | 현재값과 이전값의 차이 |
| 레벨 | Critical(min=-60) / Major(>=10dB) / Minor(>=3dB) |

### 2.4 상태 흐름

```
INIT (엔진이 티켓 생성)
  → [인지 버튼] → ACK (운용자가 장애 인지)
  → [마감 버튼] → FIN (운용자가 장애 마감)

INIT → AUTO_FIN (엔진이 자동으로 복구 판정)
```

## 3. 순단장애 — 팝업(모달) 상세

### 3.1 인지/마감 처리 (ModalRcaProcessHandling)

**활성화**: 티켓 카드의 [인지] 또는 [마감] 버튼 클릭

| 항목 | 설명 |
|------|------|
| TT번호 | 티켓 ID |
| 티켓발생시간 | 엔진이 티켓을 생성한 시간 |
| 장애발생시간 | 실제 광레벨 저하가 감지된 시간 |
| 전체 알람 수 | 해당 티켓의 총 경보 건수 |
| 인지/마감 담당자 | 현재 로그인 사용자 자동 입력 |
| 소속 팀 | 사용자 정보에서 자동 입력 (deptName) |
| 분류 | NOC 또는 EMS (사용자 정보의 agencyName) |
| 근원장애 System/Port 정보 | A측/Z측 시스템명, 포트 (그리드) |
| 인지 관련사항 (ACK) | 텍스트 입력 (최대 120자) |
| 마감 관련사항 (FIN) | 텍스트 입력 (최대 120자) |

**처리 결과**: DB에 처리 이력 저장 + 티켓 STATUS 변경

### 3.2 TT 일괄처리 (ModalRcaBatchProcessing)

**활성화**: 티켓 카드의 [TT일괄처리] 버튼 클릭

동일 캐리어 구간(trunk_name)에 속하는 **다른 티켓들**을 일괄 조회하여 한 번에 처리.

| 컬럼 | 설명 |
|------|------|
| TT No. | 티켓 ID |
| 발생일시 | 장애 발생 시간 |
| 마감일시 | 마감 처리 시간 (처리 전에는 빈값) |
| 시스템명 | A측 ROADM 시스템명 |
| 발생국소 | 캐리어 구간명 (trunk_name) |
| 상태 | INIT/ACK/FIN |
| 수리내용 | 조치 처리 시 입력한 내용 |

**버튼:**
- [인지]: 선택된 티켓을 일괄 ACK 처리
- [조치]: 조치사항 입력 모달(ModalRcaBatchFix) → HANDLING_CONTENT에 저장
- [마감]: 선택된 티켓을 일괄 FIN 처리

**참고**: 처리 후 그리드가 자동 리로드되어 상태 변경이 즉시 반영됨

### 3.3 조치사항 입력 (ModalRcaBatchFix)

**활성화**: TT일괄처리 모달의 [조치] 버튼 클릭

| 항목 | 설명 |
|------|------|
| 고장 조치확인 | 텍스트 입력 (예: "케이블 교체 후 서비스 정상 확인") |
| [확인] 버튼 | 입력값이 있을 때만 동작. 빈값이면 경고 메시지 |
| [닫기] 버튼 | 단순 닫기 (조치 프로세스 실행하지 않음) |

## 4. 예지보전 — 화면 구성

### 4.1 화면 레이아웃

```
┌──────────────────────────────────────────────────────────────┐
│  상단바: 예지보전 | Rx/Tx · Span/Gain · NTD 토글            │
├──────────┬───────────────────────────────────────────────────┤
│          │                                                   │
│  티켓    │  [2D 토폴로지]                                    │
│  목록    │  캐리어 구간의 ROADM 노드 + 광 성능 지표          │
│          │                                                   │
│  (카드   │  노드별 표시 정보:                                │
│   형태)  │  - Rx/Tx: 수신/송신 광레벨                       │
│          │  - Span Loss / Node Gain                          │
│          │  - NTD (Node Total Deviation)                     │
│          │                                                   │
└──────────┴───────────────────────────────────────────────────┘
```

### 4.2 티켓 카드 정보

| 항목 | 설명 |
|------|------|
| 티켓 번호 | ticketno |
| 발행일 (issue_date) | 예측 경보가 생성된 날짜 |
| 해제일 (clear_date) | 경보가 해제된 날짜 (null이면 진행 중) |
| 사유 (reason) | "광레벨 장기 변화 예측 경보" 또는 "Node Total Deviation 변화" |
| 캐리어명 (trunk_name) | 광 전송 구간 |
| 방향 (direction) | UP / DOWN |
| 대상 시스템 (target_sysname) | 예측 경보 대상 ROADM 장비명 |
| 장비유형 (metype) | ROADM 모델명 |
| NTD (node_total_deviation) | 노드 총 편차 (dB) |
| ROADM 수 (roadm_cnt) | 해당 구간의 ROADM 노드 수 |
| 검토 의견 수 (opinion_cnt) | 등록된 운용자 의견 건수 |
| 검토 의견 버튼 | 새 의견이 있으면 NEW 아이콘 표시 |

### 4.3 토폴로지 표시 옵션

상단 라디오 버튼으로 토폴로지 노드에 표시할 정보 선택:

| 옵션 | 설명 | 단위 |
|------|------|------|
| Rx / Tx | 수신 광레벨 / 송신 광레벨 | dBm |
| Span Loss / Node Gain | 구간 손실 / 노드 이득 | dB |
| Node Total Deviation | 노드 총 편차 (기준값 대비 변화) | dB |

## 5. 예지보전 — 팝업 상세

### 5.1 검토 의견 (ModalPredictiveReviewOpinion)

**활성화**: 티켓 카드의 [검토 의견] 버튼 클릭

운용자들이 해당 예측 경보에 대한 의견을 공유하는 게시판 형태.

**의견 목록 그리드:**

| 컬럼 | 설명 |
|------|------|
| 작성자 | 의견 작성자 이름 |
| 소속 | 담당 부서 |
| 의견 | 내용 |
| 작성 시각 | 등록/수정 시간 |
| 기능 | [수정] / [삭제] 버튼 (본인 작성글만 가능) |

**의견 입력:**
- 하단 텍스트 영역에 내용 입력 → [등록] 버튼
- 기존 의견 [수정] 클릭 시 → 하단에 내용 채워짐 → [등록] 시 수정 처리
- [삭제] → soft delete (DELETE_YN = 'Y')
- 본인 작성글만 수정/삭제 가능 (handling_user 비교)

**NEW 아이콘:**
- localStorage에 마지막 확인 시점 저장
- 확인 이후 새 의견이 있으면 티켓 카드에 NEW 표시

## 6. 데이터 용어 설명

### 6.1 광 성능 지표

| 용어 | 영문 | 설명 | 단위 |
|------|------|------|------|
| 광레벨 | Optical Power Level | 광 신호의 세기 | dBm |
| Rx | Receive | 수신 광레벨 | dBm |
| Tx | Transmit | 송신 광레벨 | dBm |
| Span Loss | Span Loss | 두 노드 간 광 신호 손실 | dB |
| Node Gain | Node Gain | 증폭기(AMP)에 의한 신호 이득 | dB |
| NTD | Node Total Deviation | 노드 총 편차. 기준값 대비 현재 광레벨 변화량 | dB |

### 6.2 장비/구간 용어

| 용어 | 설명 | 예시 |
|------|------|------|
| ROADM | 재구성 가능한 광 분기/결합기 | 서울NIA, 수원성균관대 |
| 중계기 (Repeater) | ROADM 사이에서 광 신호를 증폭하는 장비 | 중계_영동중_0801B |
| 캐리어 (Trunk) | 두 ROADM 간 광 전송 구간 | 서울NIA--수원성균관대 |
| 방향 (Direction) | 광 신호 전파 방향 | UP (A→Z), DOWN (Z→A) |
| A측 / Z측 | 캐리어 구간의 시작점 / 끝점 | A: 서울NIA, Z: 수원성균관대 |
| PTP (ptpname) | 물리 종단점. 광 포트 식별자 | SH1-S.1 |
| TID | 장비 식별자 (IP 주소) | 192.168.200.210 |
| routenum | 캐리어 내 노드 순서 | 0(시작), 1~9(중계), 10(끝) |

### 6.3 장애 관련 용어

| 용어 | 설명 |
|------|------|
| 순단장애 (MomentaryBreakoff) | 광 신호가 순간적으로 끊기거나 급격히 저하되는 현상 |
| 저광레벨 경보 | valueMax - valueMin >= 3dB 일 때 발생 |
| Critical | 최소값 = -60.0 dBm (신호 완전 소실) |
| Major | 변동폭 >= 10.0 dB |
| Minor | 변동폭 >= 3.0 dB |
| 예지보전 (Predictive Maintenance) | AI 분석을 통해 향후 장애 발생을 사전 예측 |
| NTD 변화 | Node Total Deviation이 급변하여 장비 이상 감지 |
| 광레벨 장기 변화 예측 경보 | AI 모델이 장기적 광레벨 변화 추세를 분석하여 발생 |

## 7. 운용 시 주의사항

1. **순단장애 티켓은 14일 이내**만 표시됩니다. 14일 이상 된 티켓은 화면에서 사라집니다.

2. **인지/마감 처리는 되돌릴 수 없습니다.** ACK → INIT으로 복귀하는 기능은 없으며, 인지 수정/마감 수정은 내용만 변경 가능합니다.

3. **TT 일괄처리는 동일 trunk_name 기준**입니다. 현재 선택한 티켓과 같은 캐리어 구간의 다른 티켓만 조회됩니다.

4. **예지보전 티켓의 clear_date**가 null이면 진행 중, 값이 있으면 해제된 경보입니다. 목록 상단의 토글로 진행 중/해제됨을 전환할 수 있습니다.

5. **검토 의견은 본인 작성분만 수정/삭제 가능**합니다. 다른 사용자의 의견을 수정/삭제하려고 하면 안내 메시지가 표시됩니다.

6. **현재 데이터는 시뮬레이션 상태**입니다. EMS 장비가 과거 고정 데이터를 반환하고 있어 순단장애/예지보전 모두 실시간 데이터가 아닙니다. 엔진팀 확인이 필요합니다.
  -> 코드 문제가 아니라 DB 데이터가 과거 데이터라는 뜻
