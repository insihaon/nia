# 순단장애 / 예지보전 — 시스템 구조 및 핵심 로직

> 최종 갱신: 2026-04-06

---

## 1. 시스템 개요

순단장애(MBA)와 예지보전(PM)은 ROADM 광 전송 장비의 성능 데이터를 기반으로 장애를 감지하고 예측하는 시스템이다.

```
[EMS 장비] → [linkage 모듈] → [AI 모델] → [엔진] → [DB] → [UI]
  (ROADM)     (15분 주기       (분석)      (티켓       (MBA     (Web2024)
               TL1 수집)                   생성)      스키마)
```

| 구분 | 순단장애 (MBA) | 예지보전 (PM) |
|------|--------------|-------------|
| 목적 | 회선 순단 실시간 감지 | 광모듈 장애 장기 예측 |
| 엔진 | `EngineNiaOpticalPm/preprocessor/prepro-mba` | `EngineNiaOpticalPm/predictiveMaintenance-domain` |
| 분석 주기 | 15분 (실시간) | 일별 집계 |
| 판정 기준 | `valueMax - valueMin >= 3` | NTD(Node Total Deviation) 변화 / AI 예측 경보 |
| 티켓 테이블 | `mba.tb_mba_ticket_current` | `mba.tb_roadm_predictive_maintenance_ticket` |

## 2. 데이터 파이프라인

### 2.1 데이터 수집 (공통)

```
LinkageSchdulerService (@Scheduled 15분 주기)
  → EmsMmcHdlService.pmCollectHdl()
    → EMS 장비 접속 (TL1 소켓, 61.252.53.34)
    → RTRV-PM 명령으로 ROADM 광 성능 데이터 수집
    → parsingHdlService → DB 저장
      → mba.tb_roadm_optical_performance_hist (15분 이력)
      → mba.tb_roadm_optical_performance_daily (일별 집계)
```

### 2.2 순단장애 분석

```
pmMapper.callPmPrepro() → 전처리
  → MbaAiModelPrdAmqp → RabbitMQ → AI 모델
  → nia.MbaAiModelResultIndexDirectly 큐로 결과 수신
  → MbaAiResultListener
    → LineMonitoringHdlService
      → valueMax - valueMin >= 3 이면 순단 판정
    → TicketDataService → TicketCreateService
      → mba.tb_mba_ticket_current INSERT
      → mba.tb_mba_ticket_cable INSERT
      → RabbitMQ로 UI에 TICKET_NEW 이벤트 전송
```

### 2.3 예지보전 분석

```
PmDailyService.performanceDailyHdl()
  → 일별 광성능 데이터 집계
  → AiPmDataSendService → REST API로 AI 모델에 전송 (/api/pmData)

TicketCaseOneService (NTD 변화 감지)
  → pmPreMapper.selectPmAlarmCheckList()
  → trunk_name + direction 기준 중복 체크
  → mba.tb_roadm_predictive_maintenance_ticket INSERT

TicketCaseTwoService (AI 장기 변화 예측)
  → pmPreMapper.selectWarningDateCheckList()
  → trunk_name + direction 기준 중복 체크
  → mba.tb_roadm_predictive_maintenance_ticket INSERT
```

## 6. 핵심 로직 상세

### 6.1 순단장애 인지/마감 처리

```
USER_PROCESS_MBA (UPSERT)
  → tb_mba_ticket_handling INSERT ON CONFLICT UPDATE
  → tb_mba_ticket_current STATUS 변경 (INIT → ACK → FIN)
```

| STATUS | 의미 | 기록 필드 |
|--------|------|----------|
| INIT | 초기 (엔진 생성) | - |
| ACK | 인지 | HANDLING_USER, HANDLING_DEPT, HANDLING_AGENCY, HANDLING_TIME, HANDLING_CONTENT |
| FIN | 마감 | HANDLING_FIN_USER, HANDLING_FIN_DEPT, HANDLING_FIN_AGENCY, HANDLING_ACK_TIME, HANDLING_FIN_CONTENT |
| AUTO_FIN | 자동 마감 (엔진) | - |

### 6.2 TT 일괄처리

동일 `trunk_name`의 다른 티켓을 일괄 조회하여 인지/조치/마감 처리.

```
TransientOutageList.vue [TT일괄처리] 버튼
  → ModalRcaBatchProcessing.open({ ticket })
    → SELECT_TICKET_ALARM_BATCH_LIST (동일 trunk_name 티켓)
    → [인지] → apiUserProcess('USER_PROCESS_MBA') × N건
    → [조치] → ModalRcaBatchFix → HANDLING_CONTENT에 조치내용 기록
    → [마감] → apiUserProcess('USER_PROCESS_MBA') × N건
    → loadBatchTicketList() 리로드
```

### 6.3 예지보전 검토 의견

```
PredictiveMaintenanceTicket.vue [검토 의견] 버튼
  → ModalPredictiveReviewOpinion.open(ticket)
    → SELECT_MBA_PREDICTIVE_REVIEW (의견 목록)
    → [등록] → UPDATE_MBA_PREDICTIVE_REVIEW (UPSERT)
    → [수정] → 동일 (seqnum 기준)
    → [삭제] → DELETE_MBA_PREDICTIVE_REVIEW (DELETE_YN = 'Y')
    → 글 작성자만 수정/삭제 가능 (handling_user 비교)
```

### 6.4 토폴로지 조회

순단장애/예지보전 모두 2D 토폴로지를 표시하며, 노드 선택 시 해당 노드 기준으로 알람 필터링.

| 화면 | 쿼리 | 표시 정보 |
|------|------|----------|
| 순단장애 | `SELECT_MBA_TOPOLOGY_LIST` | trunk_name, sysname, routenum |
| 예지보전 | `SELECT_PMM_TOPOLOGY_LIST` | + Rx/Tx, Span/Gain, NTD |
