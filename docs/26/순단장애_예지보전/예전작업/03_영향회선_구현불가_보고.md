## 1. 영향회선이란

순단장애/예지보전 화면에서 특정 전표를 선택하면 "영향회선 리스트" 탭이 있다. 이것은 **해당 장애 구간을 경유하는 서비스 회선 목록**을 보여주는 기능이다.

```
[장애 발생]  서울NIA ── 광케이블 ── 수원성균관대  (3dB 이상 저하)
                          │
                ┌─────────┴─────────┐
                ▼                   ▼
         E-Line 서비스 A      E-Line 서비스 B     ← 이것이 "영향회선"
         (고객사 X)           (고객사 Y)
```

| 개념 | 방향 | 질문 |
|------|------|------|
| **근원장애** | 원인 → 결과 | "장애의 원인이 무엇인가?" |
| **영향회선** | 결과 → 범위 | "이 장애로 어떤 서비스가 영향받는가?" |

## 3. 구현 불가 사유

### 사유 1: 참조 테이블 미존재

| 항목 | 값 |
|------|-----|
| 필요 테이블 | `mba.tb_ticket_ic_mst` |
| DB 존재 여부 | **미존재** |
| 원인 | 구 시스템(RCA)에서 NIA로 이식할 때 마이그레이션되지 않음 |

### 사유 2: 대체 가능한 유사 테이블 없음

현재 NIA DB에 존재하는 서비스/회선 관련 테이블을 전수 조사했다:

| 테이블 | 데이터 건수 | 전표 연결 키 | 활용 가능 여부 | 사유 |
|--------|:---------:|:----------:|:------------:|------|
| `nia.tb_tsdn_service_mst` | 23건 | 없음 | **불가** | 전표(TICKET_ID)↔서비스 매핑 키가 없음 |
| `nia.tb_tsdn_service_node_mst` | 0건 | - | **불가** | 데이터 자체가 없음 |
| `nia.tb_tsdn_service_pce_mst` | 26건 | 없음 | **불가** | ROADM 일부만 존재, 매핑 키 없음 |
| `rca.tb_rca_ticket_cable_mst` | 50,971건 | RCA 전표 전용 | **불가** | MBA 전표와 체계가 다름 |

### 사유 3: 노드명 불일치

전표 시스템과 서비스 마스터 간 노드 식별자가 달라 자동 JOIN이 불가하다:

| 시스템 | 노드 식별 방식 | 예시 |
|--------|-------------|------|
| MBA 전표 (엔진) | 한글 장비명 | `서울NIA`, `수원성균관대`, `대전ETRI` |
| T-SDN 서비스 마스터 | 영문 약어 | `SEOUL`, `SUWON`, `DAEJEON` |

매핑 테이블이 없으므로 `서울NIA = SEOUL`이라는 것을 시스템이 알 수 없다.

### 이 중에서 "영향회선" 관련 정보는?

**없다.**

위 테이블들은 모두 **장비(ROADM) 및 광케이블** 관점의 데이터다.
영향회선은 **서비스(회선)** 관점의 데이터이며, 장비↔서비스 간 매핑 정보가 DB에 존재하지 않는다.

```
[현재 보유]                              [필요하지만 없는 것]
                                        
장비 토폴로지 (ROADM A → ROADM Z)        서비스 경유 정보 (서비스 X가 어떤 ROADM을 지나는가)
광레벨 성능 (RX, TX, NTD)               장비↔서비스 매핑 (ROADM A를 지나는 서비스 목록)
전표 (어떤 구간에서 장애 발생)             노드명 매핑 (서울NIA = SEOUL)
```

## 8. 용어 정리 - 회선, 서비스, 고객의 관계

### 8.1 통신망 계층 구조

NIA 시스템에서 다루는 데이터는 크게 **물리 계층**과 **서비스 계층**으로 나뉜다. 영향회선은 이 두 계층을 연결하는 개념이다.

```
┌─────────────────────────────────────────────────────┐
│  서비스 계층 (고객이 보는 것)                          │
│                                                     │
│  고객(CUSTNAME) ── 서비스(SERVICE) ── 회선(CIRCUIT)   │
│  "A기업"           "E-Line 100M"     "전용회선 #1234" │
│                                                     │
├─────── 여기를 연결하는 것이 "영향회선" ──────────────────┤
│                                                     │
│  물리 계층 (장비가 보는 것)                             │
│                                                     │
│  장비(ROADM) ── 광케이블(TRUNK) ── 포트(PTP)          │
│  "서울NIA"       "서울-수원"         "1-1-L1"         │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 8.2 용어 정의

| 용어 | 영문 | 설명 | NIA에서의 위치 |
|------|------|------|-------------|
| **회선** (전용회선) | TRANSCIRCUIT | 고객에게 제공되는 물리적 통신 경로. 고유 번호(TRANSCIRCUITSEQ)로 식별 | 영향회선 쿼리의 주 대상 |
| **서비스** | SERVICE | 회선 위에 구성된 논리적 통신 서비스 (E-Line, E-LAN 등). 하나의 회선에 여러 서비스가 올라갈 수 있음 | `TB_TSDN_SERVICE_MST` |
| **고객** | CUSTOMER (NES) | 서비스를 이용하는 사업자 또는 기관 | 영향회선 쿼리의 CUSTNAME 필드 |

### 8.3 "서비스"와 "영향회선"의 관계

영향회선 = **특정 장애가 영향을 미치는 전용회선 목록**이다. "서비스"는 그 회선 위에 올라가는 것이므로, 실무적으로는 아래와 같이 연쇄된다:

```
[장애 발생]
    서울NIA ─── 광케이블 끊김 ─── 수원성균관대
         │
         ▼
[영향받는 회선 = 영향회선]
    전용회선 #1234 (서울→수원 경유)
    전용회선 #5678 (서울→대전 경유, 수원 경유)
         │
         ▼
[영향받는 서비스]
    E-Line 100M (회선 #1234 위에 구성)
    E-Line 1G   (회선 #5678 위에 구성)
         │
         ▼
[영향받는 고객]
    A기업 (E-Line 100M 이용)
    B기관 (E-Line 1G 이용)
```

즉 영향회선 기능은 **"이 장애로 어떤 고객이 피해를 보는가?"**를 파악하기 위한 것이다. "서비스"는 고객과 회선 사이의 중간 단위이다.

### 8.4 주석 처리된 쿼리의 컬럼이 보여주는 것

`SELECT_MBA_PMM_INFLUENCECIRCUIT_LIST`의 원본 컬럼:

| 컬럼 | 의미 | 계층 |
|------|------|------|
| `TRANSCIRCUITSEQ` | 전용회선 고유번호 | 회선 |
| `TRANSCIRCUITNAME` | 전용회선명 | 회선 |
| `LLNUM` | 전용회선 논리번호 | 회선 |
| `SCN` | 서비스 연결번호 | 서비스 |
| `CUSTNAME` | 고객명 | 고객 |
| `SVCMAIN` / `SVCSUB` / `SVCNET` | 서비스 유형 (주/부/망) | 서비스 |
| `MGMTOFFICEA` / `MGMTOFFICEZ` | A/Z측 관리국소 | 물리 위치 |
| `INSTLOCATIONA` / `INSTLOCATIONZ` | A/Z측 설치위치 | 물리 위치 |
| `INSTADDRA` / `INSTADDRZ` | A/Z측 설치주소 | 물리 위치 |

이 컬럼들은 **회선 + 서비스 + 고객 + 위치** 정보를 한 행에 모두 담고 있다. 즉 영향회선 리스트는 단순히 회선 목록이 아니라 **"어떤 고객의 어떤 서비스가, 어디에서 어디까지, 어떤 회선으로 연결되어 있는가"**를 보여주는 것이다.

### 8.5 NIA DB의 서비스 관련 테이블 (존재하지만 활용 불가)

| 테이블 | 용도 | 주요 컬럼 | 데이터 | 문제 |
|--------|------|----------|:------:|------|
| `nia.tb_tsdn_service_mst` | T-SDN 서비스 마스터 | SERVICE_ID, SERVICE_NAME, SRC_NODE_ID, DST_NODE_ID, SERVICE_TYPE | 23건 | 전표 연결 키 없음. NODE_ID가 영문 약어라 전표의 한글 장비명과 매핑 불가 |
| `nia.tb_tsdn_service_pce_mst` | 서비스 경유 경로 (PCE) | SERVICE_ID, SRC_NODE_ID, DST_NODE_ID, PCE_INDEX | 26건 | 서비스별 경유 노드. 일부 ROADM만 존재. 위와 동일한 NODE_ID 문제 |
| `nia.tb_tsdn_service_node_mst` | 서비스 노드 상세 | SERVICE_ID, NODE_ID, TP_ID, TAG | **0건** | 데이터 없음 |

이 테이블들은 `EngineNiaServer2021/NiaLinkageApplication`의 `TopologyMapper.xml`에서 T-SDN 연동 시 적재된다. 그러나:

1. **데이터가 극소량** (23건) — 전체 서비스를 대표하지 않음
2. **TICKET_ID 연결 키 없음** — 전표와 서비스를 연결할 방법이 없음
3. **CUSTNAME 없음** — 고객 정보가 포함되어 있지 않음
4. **NODE_ID 불일치** — 서비스의 `SEOUL` ≠ 전표의 `서울NIA`

따라서 이 테이블만으로는 영향회선을 구현할 수 없다.

### 8.6 참고 - DataHub의 수용고객/영향고객 개념

NIA에는 DataHub라는 별도 시스템이 있으며, 여기에 유사한 개념이 존재한다:

```
파일: DataHubMapper.xml:955~960
```

| 컬럼 | 의미 |
|------|------|
| `NEA_NES_CNT` | 수용고객 고객 수 |
| `NEA_LINE_CNT` | 수용고객 영향회선 수 |
| `NEA_SVC_CNT` | 수용고객 서비스 수 |
| `NEA_RELATION_NES_CNT` | 영향고객 고객 수 |
| `NEA_RELATION_LINE_CNT` | 영향고객 영향회선 수 |
| `NEA_RELATION_SVC_CNT` | 영향고객 서비스 수 |

DataHub는 **장비(NE) 기준**으로 수용고객/영향고객을 집계하고 있다. 이는 순단장애의 영향회선과 유사한 개념이나, DataHub는 `ndhub` 스키마의 별도 데이터를 사용하며 MBA 전표와 직접 연결되지 않는다.
