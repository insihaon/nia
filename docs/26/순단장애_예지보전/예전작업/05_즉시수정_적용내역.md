## 수정 상세

### 1. MBA-F1: WebSocket 실시간 전표 수신 파싱

```
파일: apps/Web2024/FE/src/views-nia/pages/TransientOutage/TransientOutageList.vue:403
```

```javascript
// Before
const item = message.data[0] ?? []

// After
const item = message ?? {}
```

**이유**: WebSocket으로 수신되는 MBA 전표 데이터는 `message` 자체가 전표 객체이다. `message.data[0]`은 undefined가 되어 실시간 전표 수신이 동작하지 않았음.

**복구되는 기능**: 순단장애 실시간 전표 수신 + 경보음 재생

### 2. MBA-F2: apiUserProcess 호출 SQL ID 수정

```
파일: apps/Web2024/FE/src/views-nia/pages/TransientOutage/modal/ModalRcaProcessHandling.vue:466, 486
```

```javascript
// Before (:466) - 오타 'inspecor', 미존재 SQL USER_PROCESS_INSPECOR
await apiUserProcess(param, this.isMba ? 'mba' : 'inspecor')

// After (:466) - USER_PROCESS_MBA만 존재하므로 통일
await apiUserProcess(param, 'mba')
```

```javascript
// Before (:486) - 미존재 SQL USER_PROCESS_EQUIP
await apiUserProcess(param, 'equip')

// After (:486) - USER_PROCESS_MBA로 통일
await apiUserProcess(param, 'mba')
```

**이유**: DB에 `USER_PROCESS_MBA` SQL만 존재한다. `USER_PROCESS_INSPECOR`(오타)와 `USER_PROCESS_EQUIP`는 존재하지 않아 호출 시 SQL 실행 오류가 발생했음. 원래 주석에도 `inspector 처리기능 x`라고 적혀 있었음(:475).

**복구되는 기능**: 인지/마감 처리 시 DB 저장

### 3. MBA-F3: ROOT_CAUSE_SYSNAMEZ 컬럼 추가

```
파일: apps/Web2024/BE/app-nia/src/main/resources/mapper/db2nd/transmissionFaultAnalytics.xml:1040~
```

```xml
<!-- Before -->
, CABLE.SYSNAMEA     AS ROOT_CAUSE_SYSNAMEA
, CABLE.PTPNAMEA     AS ROOT_CAUSE_PORTA
, NULL               AS ROOT_CAUSE_EQUIP_TYPEA
, CABLE.PTPNAMEZ     AS ROOT_CAUSE_PORTZ
<!-- ROOT_CAUSE_SYSNAMEZ 누락 -->

<!-- After -->
, CABLE.SYSNAMEA     AS ROOT_CAUSE_SYSNAMEA
, CABLE.SYSNAMEZ     AS ROOT_CAUSE_SYSNAMEZ     ← 추가
, CABLE.PTPNAMEA     AS ROOT_CAUSE_PORTA
, NULL               AS ROOT_CAUSE_EQUIP_TYPEA
, CABLE.PTPNAMEZ     AS ROOT_CAUSE_PORTZ
```

**이유**: Vue 템플릿(TransientOutageList.vue:99)에서 `{{ item.root_cause_sysnamez }}`를 표시하는데, SQL에서 해당 컬럼을 반환하지 않아 Z측 시스템명이 빈칸이었음.

**복구되는 기능**: 전표 상세에서 Z측 시스템명 표시

### 7. PM-F3: new_list null 안전 연산자 추가

```
파일: apps/Web2024/FE/src/views-nia/pages/PredictiveMaintenance/PredictiveMaintenanceTicket.vue:335
```

```javascript
// Before
} else if (ticket?.new_list?.filter(x => !this.readOpinionByUser[ticket.ticketno].readList.includes(x)).length > 0 ...

// After
```

**복구되는 기능**: 신규 의견 표시("NEW" 배지) 오류 방지

### 8. PM-F4: Vuex DELETE 뮤테이션 로직 수정

```
파일: apps/Web2024/FE/src/store/modules/untact.js:33~41
```

```javascript
// Before - find()는 단일 객체 반환 → 리스트가 객체로 대체됨
DELETE_TICKET_MBA_LIST: (state, value) => {
  state.mbaTicketList = state.mbaTicketList.rows.find(row => row.ticket_id === value?.ticket_id)
},
DELETE_TICKET_PMM_LIST: (state, value) => {
  state.pmmTicketList = state.pmmTicketList.rows.find(row => row.ticketno === value?.ticketno)
}

// After - filter()로 해당 항목 제외
DELETE_TICKET_MBA_LIST: (state, value) => {
  if (state.mbaTicketList?.rows) {
    state.mbaTicketList.rows = state.mbaTicketList.rows.filter(row => row.ticket_id !== value?.ticket_id)
  }
},
DELETE_TICKET_PMM_LIST: (state, value) => {
  if (state.pmmTicketList?.rows) {
    state.pmmTicketList.rows = state.pmmTicketList.rows.filter(row => row.ticketno !== value?.ticketno)
  }
}
```

**이유**: `find()`는 조건에 맞는 **단일 객체**를 반환한다. 이것을 state에 할당하면 배열이 객체로 대체되어 이후 모든 리스트 렌더링이 깨진다. `filter()`로 해당 항목만 제외하는 것이 올바른 삭제 로직이다.

**복구되는 기능**: 향후 전표 삭제 기능 추가 시 리스트 무결성 보장

### 9. MBA-F5: UPDATE_TIME → STATUS.UPDATE_TIME 수정

```
파일: apps/Web2024/BE/app-nia/src/main/resources/mapper/db2nd/transmissionFaultAnalytics.xml:1034
```

```xml
<!-- Before -->
, TO_CHAR(T.TICKET_GENERATION_TIME, 'YYYY-MM-DD HH24:MI:SS') AS UPDATE_TIME

<!-- After -->
, TO_CHAR(STATUS.UPDATE_TIME, 'YYYY-MM-DD HH24:MI:SS')        AS UPDATE_TIME
```

**DB 확인**: `mba.tb_mba_ticket_handling.update_time` 컬럼 존재 확인. 이 테이블은 SQL에서 `STATUS` alias로 LEFT OUTER JOIN됨.

**이유**: 생성시간(`T.TICKET_GENERATION_TIME`)을 갱신시간(`UPDATE_TIME`)으로 반환하고 있었음. Vue(TransientOutageList.vue:45)에서 `generation_time !== update_time`으로 갱신 여부를 판단하는데, 항상 동일값이라 갱신 표시가 불가능했음.

**복구되는 기능**: 전표 갱신 여부 판단 및 갱신 시간 표시 (처리된 전표가 없으면 NULL → v-show로 숨김 처리되어 안전)
