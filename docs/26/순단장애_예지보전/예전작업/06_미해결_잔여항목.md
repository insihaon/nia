#### PM-E2. SELECT_PMM_TOPOLOGY_LIST에서 TIME_STAMP 파라미터 무시

```
파일: PredictiveMaintenanceTicket.vue:245
  apiRcaRequest('SELECT_PMM_TOPOLOGY_LIST', { ..., TIME_STAMP: timeStamp })

파일: transmissionFaultAnalytics.xml:640~783
  → TIME_STAMP 파라미터를 WHERE 절에서 사용하지 않음
```

**증상**: 시점별 성능 데이터 조회가 아닌 전체 데이터 반환 → 잘못된 시계열 데이터

**DB 확인 결과**:

`tb_roadm_optical_performance_hist` 테이블:
- `time_stamp` 컬럼: **없음**
- `ocrtime` 컬럼: **있음** (timestamp with time zone) — 이것이 수집 시점

SQL 현재 동작:
```sql
WHERE oph.ocrtime > RCA.FC_GET_MINUS_DAY(1)  -- 최근 1일 데이터만
```

Vue에서 전달하는 `TIME_STAMP`는 특정 시점의 성능 데이터를 보기 위한 파라미터인데, SQL이 이를 무시하고 최근 1일 전체를 반환. 이후 `MAX(ocrtime)` 집계로 가장 최근 값만 사용.

**분석**: 시점별 조회가 필요하다면 `ocrtime`으로 필터링해야 함. 하지만 현재 SQL이 `MAX(ocrtime)` 집계를 사용하므로, TIME_STAMP를 범위 필터로 적용하는 것이 적절.

**해결 방안**:
- A) SQL에 `AND oph.ocrtime <= #{TIME_STAMP}::TIMESTAMP` 조건 추가 → 해당 시점까지의 최신 데이터 조회
- B) TIME_STAMP가 불필요하면 Vue에서 파라미터 제거 (현재 동작 유��)

#### PM-E3. A-Z 리스트 NTD 계산에서 node_total_deviation 누락

```
파일: PredictiveMaintenanceTicket.vue:249~257
  loadAzList(ticket)
    → apiRcaRequest('SELECT_PMM_TOPOLOGY_LIST')
    → 결과에서 인접 노드 간 NTD 차이 계산:
        const diff = Math.abs(arr[idx+1].node_total_deviation - arr[idx].node_total_deviation)
```

**증상**: NTD 차이 계산이 NaN이 되면 "조치필요 구간" 표시가 깨짐

**DB 확인 결과 — 핵심 문제 발견**:

| 테이블 | node_total_deviation | 용도 |
|--------|:-------------------:|------|
| `mba.tb_roadm_optical_performance_hist` | **없음** | SELECT_PMM_TOPOLOGY_LIST가 사용하는 테이블 |
| `mba.tb_roadm_optical_performance_daily` | **있음** (numeric) | 일별 집계 테이블 |

`SELECT_PMM_TOPOLOGY_LIST`는 `_hist` (실시간 이력) 테이블을 사용하는데, `node_total_deviation`은 `_daily` (일별 집계) 테이블에만 존재함.

`_daily` 테���블 컬럼: `collection_date, trunk_id, trunk_name, tid, sysname, ptpname, in_out, routenum, direction, value_max, span_loss, node_gain, **node_total_deviation**, ntd_variance`

**분석**: NTD는 일별 집계 시 계산되는 파생 값이므로 실시간 이력 테이블에는 없는 것이 정상. A-Z 리스트에서 NTD를 표시하려면 `_daily` 테이블에서 조회하거나, SQL에서 `_daily`를 JOIN해야 함.

**해결 방안**:
- A) `SELECT_PMM_TOPOLOGY_LIST`에 `_daily` 테이블 LEFT JOIN 추가하여 `node_total_deviation` 포함
- B) A-Z 리스트 전용 SQL을 `_daily` 테이블 기반으로 별도 작성
- C) Vue에서 `_hist` 데이터의 `value_max` 차이로 NTD를 근사 계산 (정확도 저하)
