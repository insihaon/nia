## 3. 서브프로젝트 통합 이력 (git subtree add)

현재 레포는 **git subtree add** 명령으로 여러 원본 레포를 합친 통합 레포이다.
subtree merge 커밋이 존재하며, 원본 이력이 로컬 git object에 포함되어 있다.

### 3.1 subtree 통합 내역

| # | subtree 경로 | 원본 커밋 해시 | 원본 커밋 수 | 최초 커밋 날짜 | subtree add 커밋 |
|---|-------------|--------------|-------------|--------------|----------------|
| 1 | `apps/Web2024/` (구 `Front_web_2024/`) | `5eb858de` | **781** | 2024-01-15 | `09ba0426` |
| 2 | `apps/EngineNiaServer2021/` | `b25ebdcf` | **221** | 2021-08-04 | `99be41f3` |
| 3 | `apps/EngineNiaOpticalPm/` | `01d81975` | **4** | - | `03cef6b9` |
| 4 | `apps/EngineNiaIpSdnOproute/` | `8976f850` | **3** | - | `f8c0f3ba` |
| 5 | `legacy/2021NIA/` | `e5a024c2` | **1,023** | 2021-08-04 | `365a2b25` |

### 3.2 원본 이력 보존 상태

**현재 상태: 로컬 git object에 원본 이력이 보존되어 있음**

- subtree add 시 `--squash` 없이 머지했기 때문에, 원본 커밋들이 `.git/objects/`에 존재
- `git log <원본커밋해시>` 로 원본 이력 조회 가능
- 단, 이 이력들은 `main` 브랜치의 first-parent 그래프에는 나타나지 않음
- **새 레포로 push할 때 이 orphan 커밋들이 누락될 위험이 있음** (중요!)

### 3.3 이력 소실 위험 지점

| 위험 | 설명 | 심각도 |
|------|------|--------|
| `git clone --depth` | shallow clone 시 subtree 원본 이력 소실 | 높음 |
| `git push` (단순) | reachable 커밋만 push되므로, subtree merge 부모가 아닌 orphan 커밋은 push되지 않을 수 있음 | 높음 |
| `git gc --prune` | unreachable object 삭제 시 원본 이력 소실 | 높음 |
| subtree 원본 레포 삭제 | 원본 레포가 삭제되면 복구 불가 | 높음 |

## 4. 이력 복구 전략 옵션 비교

### 옵션 A: 현재 상태 그대로 push (단순 mirror)

```bash
git remote add new-origin https://github.com/insihaon/nia.git
git push new-origin --mirror
```

- **장점**: 가장 간단, 모든 object를 그대로 전송
- **단점**: subtree 원본 이력이 orphan 상태로 존재 → 가시성 낮음
- **이력 보존**: `--mirror`는 모든 refs + objects를 전송하므로 기술적으로 보존됨
- **주의**: `--mirror`는 모든 ref를 덮어쓰므로 신규 레포가 비어있어야 함

### 옵션 B: subtree 원본 이력을 브랜치로 태깅

```bash
# 원본 이력을 명시적 브랜치/태그로 만들어 reachable하게 만듦
git branch archive/web2024-original 5eb858de
git branch archive/engine2021-original b25ebdcf
git branch archive/optical-pm-original 01d81975
git branch archive/ip-sdn-original 8976f850
git branch archive/legacy-2021-original e5a024c2
git branch archive/chatbot-tool-original 5e6b6f48
git branch archive/topology2d-original bb0d53b3
```

- **장점**: 원본 이력이 명시적 브랜치로 보존되어 `git log`로 쉽게 조회 가능
- **장점**: 일반 `git push --all`로도 이력이 전송됨
- **단점**: 추가 브랜치가 7개 생김 (archive/ prefix로 정리 가능)
- **이력 보존**: 완벽 — 각 원본 프로젝트의 전체 이력을 독립 브랜치로 보존

### 옵션 C: git filter-repo로 이력 재작성

- **장점**: 깨끗한 이력
- **단점**: 커밋 해시 변경, 복잡, 위험
- **권장하지 않음**: 이력 보존 목적에 반함

---

## 5. 권장 전략

**옵션 B (subtree 원본 이력을 archive 브랜치로 태깅) + mirror push**

이유:
1. 원본 서브프로젝트 이력이 명시적 브랜치로 보존되어 가시성 확보
2. `--mirror` push로 모든 object가 확실히 전송됨
3. 커밋 해시가 변경되지 않아 기존 참조가 유효
4. 작업이 비교적 단순하고 되돌리기 쉬움
